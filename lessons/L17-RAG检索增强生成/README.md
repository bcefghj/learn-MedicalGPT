# L17 RAG 检索增强生成（Retrieval Augmented Generation）

> **一句话精髓：**「让模型先查资料再回答，准确率翻倍。」

本课从「为什么需要 RAG」讲到「向量检索 + 生成」如何落地，并精读 MedicalGPT 中的 `chatpdf.py`：它不是一个玩具 Demo，而是一套可直接改造的 **医疗问答 RAG 脚手架**。

## 一、什么是 RAG（Retrieval Augmented Generation）

### 1.1 一句话定义

**RAG = 先从外部知识库检索相关片段，再把片段作为上下文交给大模型生成答案。**

### 1.2 直觉类比

```
   闭卷考试 vs 开卷考试
   =====================

   纯大模型：闭卷（只靠参数记忆）
   RAG：     开卷（允许带「指定资料页」）
```

模型仍然负责「组织语言、推理、总结」，但**事实锚点**被尽量固定在检索到的文本上。

### 1.3 RAG 在系统架构中的位置

```
          用户问题 query
                |
                v
        +---------------+
        |   检索模块     |  <-- 向量相似度 / BM25 / 混合
        +---------------+
                |
                | top-k 文档片段 context
                v
        +---------------+
        |   拼接 Prompt  |  <-- 「已知信息」+「问题」
        +---------------+
                |
                v
        +---------------+
        |   LLM 生成     |
        +---------------+
                |
                v
             回答 answer
```

## 二、为什么需要 RAG：时效性、覆盖面与幻觉

### 2.1 大模型的知识边界（简化模型）

```
   参数记忆（训练时见过）
   ----------------------

   强：通用语言模式、常见百科「平均知识」
   弱：最新指南、本院内部规章、未公开论文细节、私有病历（且涉及合规）
```

### 2.2 「幻觉」在医疗场景为什么更危险

医疗问答往往要求：

- **可溯源**：这句话依据哪份指南、哪一页？
- **可更新**：诊疗路径随指南迭代而变化。
- **可拒绝**：资料不足时应明确说「无法从给定信息得出」，而不是编造。

RAG 不是万能药，但它把系统行为从「纯生成」推向「**有依据的生成**」，这是工程上的重要一步。

### 2.3 什么时候 RAG 特别划算

| 场景 | 为什么 RAG 合适 |
|------|-----------------|
| 指南/说明书/教材问答 | 文本结构清晰，检索命中率高 |
| 医院内部知识库 | 数据私有，不能全靠预训练「记住」 |
| 药品说明书 QA | 需要引用原文，减少胡编 |

## 三、RAG 工作流程：检索 → 增强 → 生成

### 3.1 三步拆解

```
   Step A: 检索（Retrieval）
   -------------------------

   文档 D --> 切 chunk --> 建索引（向量 / 倒排）
                |
   query -----> 相似度排序 -----> topn chunks

   Step B: 增强（Augment）
   -----------------------

   prompt = f(指令模板, context=chunks, query=query)

   Step C: 生成（Generation）
   ---------------------------

   LLM(prompt) --> answer
```

### 3.2 chunk 是什么？为什么重要？

**chunk** 是把长文档切成「适合嵌入与检索」的小段。太小：上下文不足；太大：噪声多、embedding 语义被稀释。

```
   长文档
   ========================..................

   chunk1 | chunk2 | chunk3 | chunk4
   -----
   重叠 overlap 可避免边界句子被切断
```

## 四、向量数据库基础（从 0 到能聊面试）

### 4.1 向量数据库解决什么问题

当你有 **百万级 chunk** 时，暴力遍历计算相似度太慢。向量库提供：

- **存储向量**
- **近似最近邻搜索（ANN）**
- （可选）**元数据过滤**（科室、版本号、文档类型）

### 4.2 常见产品谱系（了解即可）

```
   向量检索存储
   ------------

   轻量：FAISS、Milvus Lite、本地文件缓存
   工程：Milvus、Qdrant、Weaviate、pgvector
```

MedicalGPT 的 `chatpdf.py` 走的是 **similarities 库 + 可选落盘 embedding** 的路线，更适合「脚本级原型 → 再换工业向量库」。

### 4.3 面试常问：「向量库和 Elasticsearch 区别？」

**简短答法：**

- ES 强项是 **关键词检索** 与复杂过滤；向量检索是其一能力之一。
- 纯向量场景有时用 **FAISS/Qdrant** 更直接；混合检索常 **BM25 + 向量** 各取所长。

## 五、Embedding 模型：把文本变成「可检索的坐标」

### 5.1 直觉

把句子映射到高维向量空间：**语义相近**的句子，向量距离更近。

```
   "高血压如何用药"  ·····················>  v1
   "血压高吃什么药"  ·····················>  v2

   v1 与 v2 在空间中更接近（理想情况）
```

### 5.2 MedicalGPT 默认用的相似度模型

在 `chatpdf.py` 中，默认组合里包含：

- **BertSimilarity**：基于句向量模型（默认 `shibing624/text2vec-base-multilingual`），偏语义。
- **BM25Similarity**：偏关键词匹配。
- **EnsembleSimilarity**：加权融合，缓解「纯向量漏关键词」或「纯 BM25 不懂同义」的问题。

**面试亮点：** 你能说「我不是只信向量，我用 **混合检索** 提升鲁棒性」。

### 5.3 选型建议（实践向）

| 需求 | 倾向 |
|------|------|
| 中英文混杂、口语化 | 多语言句向量 + BM25 混合 |
| 强术语、药品名精确匹配 | 提高 BM25 权重或做词典扩展 |
| 极长文档 | 调 chunk_size / overlap，必要时层次摘要 |

## 六、MedicalGPT 中的 RAG 实现：`chatpdf.py` 精读

> 下列模块划分与仓库文件一致：[chatpdf.py](https://github.com/shibing624/MedicalGPT/blob/main/chatpdf.py)

### 6.1 总览：类 `ChatPDF` 的职责

```
   ChatPDF
   =======

   __init__
     ├─ 选择 device（cuda / mps / cpu）
     ├─ SentenceSplitter 切分
     ├─ 初始化 sim_model（默认 Ensemble：Bert + BM25）
     ├─ _init_gen_model 加载生成模型（可选 LoRA / 量化）
     └─ add_corpus  ingest 文档

   predict / predict_stream
     ├─ most_similar(query, topn)
     ├─ 组装 RAG_PROMPT（已知内容 + 问题）
     └─ stream_generate_answer 流式生成
```

### 6.2 提示模板 `RAG_PROMPT`（工程关键点）

仓库中的中文模板核心思想：

- 明确要求 **基于已知信息** 回答。
- 要求 **无法回答时拒绝编造**（用固定中文句式约束）。

**面试考点：** 提示词工程是 RAG 的「第二道闸」，和检索同样重要。

### 6.3 `SentenceSplitter`：中英分轨

```
   split_text
      |
      +-- 含中文？ --是--> jieba 切词 + 句末标点规则
      |
      否 --> 英文按句子切分 + chunk 合并
```

**意义：** 中文没有天然空格，**jieba** 帮助避免「硬切字符」导致语义断裂。

### 6.4 `add_corpus`：多格式 ingest

支持：

- `.pdf`（PyPDF2 抽文本）
- `.docx`
- `.md`（转 HTML 再抽纯文本）
- `.txt`

流程：

```
   file -> 纯文本行列表 -> join -> split_text -> chunks -> sim_model.add_corpus
```

### 6.5 `predict`：检索 + 拼接 + 生成

核心逻辑（语义描述）：

1. 若语料库非空：`most_similar` 取 topn。
2. 若无结果：返回「没有提供足够的相关信息」。
3. 将引用片段编号：`[1] "..."`。
4. `context_str` 与 `RAG_PROMPT` 拼接，再送入生成模型。

**注意：** `context_str` 会截断到 `context_len - len(RAG_PROMPT)`，防止 prompt 过长。

### 6.6 生成侧：可选 LoRA 与量化

`_init_gen_model` 中：

- `AutoModelForCausalLM.from_pretrained(..., load_in_4bit/int8, device_map=...)`
- 若提供 `peft_name`：`PeftModel.from_pretrained` 叠加适配器

**面试话术：** 「RAG 的检索与生成是解耦的：我可以换 embedding、也可以换生成底座或挂 LoRA。」

### 6.7 嵌入缓存：`save_corpus_emb` / `load_corpus_emb`

若相似度模型支持 embedding 落盘，可避免每次启动重建索引，适合 **大语料 + 反复调试生成参数** 的阶段。

## 七、RAG 在医疗场景的价值与边界

### 7.1 价值（工程叙事）

- **可追溯**：回答可附带引用片段（`reference_results`）。
- **可更新**：换库/加文档不必重训 7B 模型。
- **成本可控**：小模型 + 好检索有时胜过盲目上大模型。

### 7.2 边界（合规与产品）

```
   医疗 RAG 不等于医疗诊断
   =======================

   RAG 提升的是「有据可查的文本生成」
   不自动满足：临床安全、责任归属、隐私合规
```

**面试加分：** 主动谈 **免责声明、人工审核、权限控制、日志审计**。

## 八、RAG vs 微调：何时选哪个

### 8.1 对照表

| 维度 | RAG | 微调（SFT 等） |
|------|-----|----------------|
| 知识更新 | 快（换文档） | 慢（重训或增量） |
| 语言风格/指令遵循 | 依赖底座 + 提示词 | 可专门塑造 |
| 计算成本 | 检索+推理 | 训练贵 |
| 隐私合规 | 可把知识放内网库 | 训练数据治理要求高 |

### 8.2 组合策略（工业界常见）

```
            用户请求
               |
               v
      +----------------+
      | 需要强事实？   |
      +----------------+
         |            |
        是           否
         |            |
         v            v
    RAG 优先      直接生成 / 轻提示
         |
         +------> 仍不够稳？
                     |
                     v
                 SFT/DPO
              「说话方式 + 安全边界」
```

**结论句（可背）：** 「**事实**交给 RAG，**行为**交给对齐微调。」

## 九、面试高频题：RAG 的优缺点？和微调怎么配合？

### 9.1 RAG 的优点（答 4 点足够）

1. **可更新知识**：文档迭代快。
2. **可溯源**：引用检索片段。
3. **成本结构清晰**：不必为每条知识扩容参数。
4. **可解释性更强**：至少能回答「参考了哪段」。

### 9.2 RAG 的缺点（一定要诚实）

1. **检索错 = 全文错**：garbage in, garbage out。
2. **chunk 与排序上限**：边界信息、跨段推理仍难。
3. **系统复杂度上升**：索引、解析 PDF、时延、监控。
4. **提示词攻击/注入**：恶意文档可误导生成（要有过滤与权限）。

### 9.3 与微调配合（标准答案骨架）

- 用 **SFT/DPO** 让模型学会：**如何引用**、**如何拒答**、**如何结构化输出**（例如先列要点再给建议）。
- 用 **RAG** 提供：**最新指南原文**、**药品说明书细节**。

## 十、动手指南：如何跑 `chatpdf.py`（思路）

仓库依赖提示（见文件头注释）：

```bash
pip install similarities PyPDF2 -U
# docx/markdown 相关按需要安装
```

命令行入口（示例来自 `__main__` 语义）：

```bash
python chatpdf.py \
  --gen_model <你的底座> \
  --corpus_files data/rag/medical_corpus.txt \
  --chunk_size 100 \
  --chunk_overlap 5
```

**建议你做的实验：**

1. 把 `topn` 从 3 调到 8，观察「引用变多」是否提升还是引入噪声。
2. 调 `chunk_size`，记录命中率与回答质量变化。
3. 换成你自己的指南 txt，看拒答模板是否按预期触发。

### 渐进复习 1：RAG 三字母分别代表什么？

- **R**：Retrieval，检索。
- **A**：Augmented，把检索结果注入上下文。
- **G**：Generation，由 LLM 生成最终回答。

### 渐进复习 2：`ChatPDF` 里谁负责「检索」？

`self.sim_model.most_similar(query, topn=topn)`。

### 渐进复习 3：谁负责「生成」？

`self.gen_model.generate`（在独立线程里配合 `TextIteratorStreamer`）。

### 渐进复习 4：为什么需要 `history`？

用于多轮对话拼装；RAG 场景下 `predict` 会清空并写入当前拼接后的 prompt。

### 渐进复习 5：医疗场景最危险的错误类型是什么？

**无依据的肯定句**（把猜测说成事实）。RAG + 拒答模板是降低概率的手段之一。

### 渐进复习 6：RAG 能否替代 SFT？

一般不能。SFT 解决「怎么说、怎么遵循格式」；RAG 解决「说什么事实」。二者互补。

### 渐进复习 7：为什么要编号引用片段？

方便前端展示引用，也方便你调试「模型到底读了哪段」。

### 渐进复习 8：Embedding 模型越大越好吗？

不一定；越大越慢，且领域不匹配时未必更好。

### 渐进复习 9：为何 MedicalGPT 默认 multilingual 向量模型？

医疗数据常夹杂英文药名、缩写，多语言模型更稳。

### 渐进复习 10：你如何在简历里写 RAG 项目？

写清：**语料规模**、**检索方案**、**延迟**、**指标**（命中率、人工抽检合格率），避免只写「用了 RAG」。

## 附录 K：RAG 效果怎么评（工程向，面试加分）

RAG 不能只凭「感觉更准」，建议至少准备 **离线检索指标 + 端到端抽检** 两层叙事。

```
   评测分层
   ========

   检索层                生成层
   --------              --------
   Hit@k / MRR          人工：有据与否、是否胡编
   nDCG（若有多级相关）  LLM-as-judge（谨慎，需对照人工）
```

| 指标 | 含义（口语） | 适用 |
|------|----------------|------|
| **Recall@k** | 正确答案所在 chunk 是否出现在前 k 条检索结果里 | 查「检索是否漏掉关键段」 |
| **MRR** | 第一个相关结果排多前 | 整体排序质量 |
| **端到端准确率** | 固定题库上「回答是否正确/可接受」 | 业务汇报 |

**与 `chatpdf.py` 结合的调试做法：**

1. 准备 20～50 条带标准引用段落 id 的问答（院内可用脱敏场景题）。
2. 打印 `most_similar` 的 topn 标题/片段，人工标是否含答案依据。
3. 再单独看生成是否 **越权发挥**（检索里没有却写得很肯定）。
