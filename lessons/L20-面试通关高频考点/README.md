## 一、项目概述类（10 题）

### 1. 介绍一下你做的 MedicalGPT 项目

- **30 秒要点**：
  - 目标：中文医疗 **领域适配**（非泛泛「用大模型」）。
  - 路线：**PT/CPT（可选）→ SFT → 偏好对齐（DPO 等可选）**。
  - 结果：至少 **1 个指标 + 数据规模 + 硬件**。
- **参考答案（口播）**：
  我做的是一个面向中文医疗场景的大模型领域适配项目，整体参考了开源 MedicalGPT 的训练流水线：先用领域语料做可选的增量预训练，让模型熟悉医学语言与知识分布；再用指令数据做有监督微调，让模型学会按对话模板回答问题；如果资源允许，还会用偏好对齐进一步改善安全性与回答风格。基座我选的是【某开源模型】，参数规模【N】，在【算力条件】下主要用 LoRA/QLoRA 训练。评测上我自建了【题型说明、N 条】测试集，相比微调前【指标名】大约提升了【x%】。需要说明的是这是内部测试结果，主要用于相对比较和迭代，不是对外宣称超越某商用模型。
- **常见追问与应对**：
  - 「和直接调用 ChatGPT 有什么区别？」→ **数据域可控、可私有化部署、成本结构不同、可做合规审计**。
  - 「创新点在哪？」→ 优先讲 **数据构建、评测协议、工程稳定性**，慎吹「算法独创」。
  - 「训练了多久？」→ 报 **真实 epoch / GPU 小时** 与是否早停。
- **易错点与加分表述**：
  **易错**：把 **调 API** 写成 **训练**。**加分**：能讲清 **每个阶段的 loss 对象与数据形态**。

### 2. 你在项目中具体负责什么？

- **30 秒要点**：
  - 用 **动词 + 产出物**：脚本、数据集版本、ckpt、报告。
  - 分 **数据 / 训练 / 评测** 至少两层，不要一锅粥。
- **参考答案（口播）**：
  我在项目里主要负责【数据/训练/评测里选一到两块】。数据侧我交付了从原始语料到训练 JSONL 的清洗与格式转换，包括去重规则、字段校验、train/val/test 划分，以及数据版本记录。训练侧我负责在【基座】上配置 LoRA/QLoRA，做了学习率与 rank 的对比实验，并维护训练日志与 checkpoint 选择标准。评测侧我搭建了【N】条内部题集，结合自动指标与抽样盲评，把结论写进实验报告。和同学的边界是【接口说明】，我这边输出【权重与配置】，对方负责【部署/前端】。
- **常见追问与应对**：
  - 「你最深的坑在哪？」→ 准备 **模板不一致 / 显存 / 对齐不稳定** 之一，讲 **定位过程**。
  - 「有没有代码贡献？」→ 指向 **具体脚本或模块**（据实）。
- **易错点与加分表述**：
  **易错**：全是「我们」却没有可验证个人产出。**加分**：能拿出 **git/配置/commit** 级别的细节。

### 3. 为什么选择这个基座模型？

- **30 秒要点**：
  - **中文与医疗**、**License**、**生态**、**自家小实验** 四件事至少讲三件。
  - 最后补一句 **代价**（显存/速度/许可限制）。
- **参考答案（口播）**：
  选型我主要考虑四方面：第一是中文与医学相关能力在公开评测与我们的 pilot 上表现够用；第二是许可证允许我们【研究/POC/商用】；第三是 HuggingFace 与社区工具链成熟，便于用 Transformers、PEFT、DeepSpeed 快速迭代；第四是我在【同数据子集】上对比过【候选 A/B】，【某模型】在【指标】上更稳。代价是【更大显存/更高延迟/某能力弱】，所以后续用 LoRA/QLoRA 和截断策略补齐工程约束。
- **常见追问与应对**：
  - 「更大模型是不是更好？」→ **边际收益递减**、**迭代周期**、**推理成本**。
  - 「有没有试过 MoE/闭源 API？」→ 如实答 **成本、合规、可控性**。
- **易错点与加分表述**：
  **易错**：只答「排行榜高」。**加分**：有 **A/B 表格或曲线** 的口头描述。

### 4. 项目的技术架构是什么？

- **30 秒要点**：
  - **数据层 / 训练层 / 评估层 / 服务层（可选）** 四层叙述。
  - 强调 **模块契约**：schema、配置、产物。
- **参考答案（口播）**：
  整体可以分成四层：数据层负责语料、指令与偏好数据的存储与版本管理，统一成 JSONL schema，并记录 hash；训练层基于 PyTorch + HuggingFace Transformers，必要时加 DeepSpeed，使用 LoRA/QLoRA 做参数高效微调，并保存 checkpoint 与训练配置 YAML；评估层离线跑自动指标，同时有人工盲评表与 bad case 归档；如果做到演示或内网服务，会有导出合并权重、用 vLLM 或 TGI 提供推理 API，并在网关侧做限流与审计。层与层之间通过 **固定路径的配置 + 数据 manifest** 交接，避免「口头约定」。
- **常见追问与应对**：
  - 「数据怎么进 Trainer？」→ **Dataset、collator、模板函数**。
  - 「如何保证训练推理一致？」→ **同一 chat_template**。
- **易错点与加分表述**：
  **易错**：只画框图没有接口。**加分**：提到 **可观测性**（日志、ckpt 命名规范）。

### 5. 数据是怎么来的？

- **30 秒要点**：
  - **来源类别** + **合规** + **规模** + **质控**。
  - 医疗务必提 **脱敏/授权**。
- **参考答案（口播）**：
  数据主要来自【公开医疗百科/指南摘要/考试题库/开源对话集/合成数据（如有）】。合规上【仅使用公开授权数据 / 院内数据已脱敏并通过伦理流程——据实描述】。规模上纯文本大约【x GB 或 x 篇】，指令数据【N 万条】，偏好数据【M 对】。质控包括统一编码、去重、长度过滤、脏符号与乱码规则，以及对高风险字段做抽检；合成数据如果有，会经过【校验器/规则/人工 spot check】降低错误累积。
- **常见追问与应对**：
  - 「有没有数据泄露风险？」→ **划分方式、去重粒度、文档级隔离**。
  - 「偏见怎么处理？」→ **分层采样、对抗评测集**。
- **易错点与加分表述**：
  **易错**：来源含糊。**加分**：能解释 **为什么这个来源适合你的任务**。

### 6. 项目的最终交付物是什么？

- **30 秒要点**：
  - **可复现包**、**权重**、**评测报告**、**版本说明**、可选 **服务镜像**。
- **参考答案（口播）**：
  交付物包括：可复现的训练配置与启动脚本、数据与模型版本说明（含 hash 或 commit）、SFT 与（如有）对齐后的权重或 LoRA adapter、评测报告（指标定义、结果表格、典型案例与 bad case 分类）、以及实验记录方便他人复跑。若做到演示，还会有推理侧 Dockerfile 或 API 说明与简单的压测数据。验收标准上我们内部约定【指标阈值】+【业务侧抽样通过】+【回归集不劣化】。
- **常见追问与应对**：
  - 「如何保证别人能复现？」→ **环境版本、随机种子、命令行、数据 manifest**。
  - 「线上呢？」→ **灰度、回滚、监控**。
- **易错点与加分表述**：
  **易错**：只交 ckpt 无报告。**加分**：**模型卡/数据卡要点**。

### 7. 你和开源 MedicalGPT 有什么相同与不同？

- **30 秒要点**：
  - **同**：阶段划分与 HF 生态。
  - **异**：数据域、基座、超参、评测、是否 RAG。
  - **加分**：对应到 **脚本/模块**（据实）。
- **参考答案（口播）**：
  相同点主要是训练阶段划分一致：都是 PT（可选）、SFT、再到 RLHF/DPO 等对齐思路，并且大量使用 HuggingFace 生态组件。不同点在于我的数据域是【具体范围】，基座换成【模型】，超参按【显存与数据量】重新扫过；评测我没有完全依赖公开榜，而是自建【题型】更贴近目标场景；此外我【做了/没做】RAG，因为【知识更新/引用需求】。如果面试官关心代码，我主要阅读并对照的是【例如 supervised_finetuning / dpo_training 等模块】来理解 loss 与数据格式。
- **常见追问与应对**：
  - 「你改了哪些代码？」→ 如实列举 **配置、数据管线、评测脚本**。
  - 「为什么不 fork 直接 PR？」→ 若有内部数据可解释 **私有分支**。
- **易错点与加分表述**：
  **易错**：声称完全一致却无差异。**加分**：**指出一次你定位的 bug 或坑**。

### 8. 训练流水线如何保证可复现？

- **30 秒要点**：
  - **版本钉死**：torch、transformers、deepspeed、CUDA。
  - **数据 hash**、**git commit**、**命令行记录**。
  - **诚实**：分布式下 bitwise 完全一致难，但趋势应稳。
- **参考答案（口播）**：
  可复现我主要抓四件事：软件版本用 requirements 或 conda 环境导出钉死；代码用 git commit id 关联每次实验；数据用 manifest 记录文件列表与 hash；训练命令与超参完整保存在实验日志里。随机性上设置种子，并说明分布式下由于数据顺序与浮点累加 order，可能无法 bitwise 一致，但同一配置重复跑结论应同一方向。若结论敏感，我会用 **多 seed 或重复实验** 说明方差。
- **常见追问与应对**：
  - 「你实际遇到过复现问题吗？」→ 准备 **某次 loss 曲线差异** 的排查故事。
  - 「DeepSpeed 呢？」→ **配置文件与 hostfile**。
- **易错点与加分表述**：
  **易错**：空谈「完全复现」。**加分**：**实验追踪工具**（W&B、MLflow、甚至 CSV）。

### 9. 项目中最大的风险是什么？如何缓解？

- **30 秒要点**：
  - **合规与安全**、**幻觉**、**工程可用性** 各一条。
  - 每条配 **缓解手段**。
- **参考答案（口播）**：
  最大风险我列三类：第一是合规与产品风险，医疗场景不能输出可误导读者的确定性诊断或用药指令，所以产品侧要有免责声明、拦截策略与人工复核流程；第二是模型幻觉与知识过时，缓解上除了训练对齐，还可以引入 RAG 与引用来源，并对高风险问题模板化拒答；第三是工程风险，比如训练中断与数据污染，我们用 checkpoint 续训、异常告警，并对数据做多层过滤与抽检。
- **常见追问与应对**：
  - 「出过事故吗？」→ 诚实：**POC 阶段如何避免**。
  - 「谁负责安全策略？」→ **产品/法务/医生** 协作（据实）。
- **易错点与加分表述**：
  **易错**：只谈技术不谈合规。**加分**：**具体拦截规则示例**（可匿名）。

### 10. 若用一句话向非技术人员汇报，你怎么说？

- **30 秒要点**：
  - **价值 + 手段 + 数字**；避免术语轰炸。
  - 用「补课」「纠偏」类比喻可加分。
- **参考答案（口播）**：
  我会说：我们在通用大模型基础上，用医疗相关的书和题库给它「补课」，再用偏好训练把回答方式「纠偏」，让它在【导诊/问答】场景更专业、更少瞎编；在我们内部【N】条测试里，【指标】大概提升了【x%】，后续还会继续用真实反馈迭代。
- **常见追问与应对**：
  - 「要花多少钱？」→ **GPU 小时粗算** 或 **租用成本**（若清楚）。
  - 「多久上线？」→ **阶段划分**。
- **易错点与加分表述**：
  **易错**：说「超越 GPT」。**加分**：强调 **人机协同** 与 **边界**。

## 二、预训练类（8 题）

### 11. PT（续训/增量预训练）和 SFT 有什么区别？

- **30 秒要点**：
  - **数据形态**：连续文本 vs 指令对话。
  - **优化目标**：LM vs 任务模仿（常 mask prompt）。
  - **能力**：补分布/知识 vs 学格式与遵循。
- **参考答案（口播）**：
  PT 或 CPT 通常是在大规模连续文本上做因果语言建模，让模型更熟悉领域语言与统计知识；数据一般是纯文本段落。SFT 则是在指令或对话数据上训练，让模型学会按人类期望的格式与策略回答，通常只对 assistant 部分算 loss。简单讲，PT 更像「读专业书」，SFT 更像「按标准答案练答题」。是否做 PT 取决于基座领域差距与我们有没有足够干净的域内语料。
- **常见追问与应对**：
  - 「只做 SFT 不行吗？」→ **可以**，看领域差距与数据。
  - 「PT 会让模型更会聊天吗？」→ **不必然**，聊天靠 SFT/对齐。
- **易错点与加分表述**：
  **易错**：把 PT 当成万能。**加分**：提 **seen tokens 与 PPL**。

### 12. 增量预训练（CPT）在什么情况下是必要的？

- **30 秒要点**：
  - **必要**：域内词/概念弱、语料多、SFT 数据少。
  - **非必要**：基座已强或 RAG 更合适。
- **参考答案（口播）**：
  当基座在目标领域语言与术语上明显弱，并且我们手里有大规模高质量域内纯文本时，CPT 往往能把后续 SFT 的上限抬高，表现为同样指令数据下收敛更快或下游更好。若基座已经很强、主要问题是「不会按格式答」或知识更新频繁，可能直接 SFT 或 RAG 更划算。CPT 也有风险：语料过偏或 epoch 过大可能带来遗忘，所以要监控通用集。
- **常见追问与应对**：
  - 「你怎么判断要做 CPT？」→ **pilot + 消融**：无 CPT vs 有 CPT。
  - 「CPT 数据从哪来？」→ **合规来源 + 去重**。
- **易错点与加分表述**：
  **易错**：为了「显得专业」强行 CPT。**加分**：**混合通用语料比例**。

### 13. 预训练数据一般怎么处理？

- **30 秒要点**：
  - **去重 → 质量过滤 → 合规 → 分块 → shuffle**。
  - 医疗：**PII**、结构保留。
- **参考答案（口播）**：
  典型流水线是先做文档级或近重复去重，再用规则与简单分类器去掉乱码、异常语言、过短过长样本；随后做隐私与合规处理，医疗场景尤其注意个人信息；再按 max length 切块或做 packing 提升吞吐，最后 shuffle。医疗文本有时会保留标题层级结构，避免把指南切碎到语义不完整；也会做分层采样，防止某个站点或科室 dominate。
- **常见追问与应对**：
  - 「packing 注意什么？」→ **attention mask 隔离**。
  - 「如何防数据污染？」→ **测试集隔离与 hash**。
- **易错点与加分表述**：
  **易错**：只谈去重不谈合规。**加分**：**MinHash/LSH** 等（若用过）。

### 14. 预训练训练了多少个 epoch？怎么定？

- **30 秒要点**：
  - 报 **真实值** + **seen tokens**。
  - **常见 1～3**；看域内 PPL 与通用集遗忘。
- **参考答案（口播）**：
  我这次 CPT 跑了【真实 epoch 数】，对应大约【seen tokens】。定 epoch 时主要看域内验证集 PPL 是否还在下降，同时监控通用能力集是否明显掉点；如果域内收益边际变小而遗忘加重，就提前停。医疗场景我倾向保守：小步多评估，而不是一次训很多 epoch。
- **常见追问与应对**：
  - 「过大怎么办？」→ **遗忘、过拟合某语料风格**。
  - 「学习率呢？」→ **常比 SFT 更谨慎 + warmup+余弦**。
- **易错点与加分表述**：
  **易错**：拍脑袋 epoch。**加分**：**学习率与数据量联动**的解释。

### 15. PT 会不会导致灾难性遗忘？会，怎么缓解？

- **30 秒要点**：
  - **会**：域语料太强、LR 太大、epoch 太多。
  - **缓解**：混合通用、小 LR、短 CPT、后续混合 SFT。
- **参考答案（口播）**：
  会。若 CPT 语料非常窄且训练过猛，模型可能在通用任务上掉点，表现为通用问答变弱或格式异常。缓解思路包括：在 CPT 中混入一定比例的通用语料；使用较小学习率与较短训练；训练后做带通用数据的 SFT replay；以及对齐阶段用 KL 类思想约束不要离参考模型太远（在 RLHF/DPO 语境）。我们会同时跟踪 **域内集** 与 **域外集**。
- **常见追问与应对**：
  - 「你怎么测遗忘？」→ **固定通用 benchmark + 业务题**。
  - 「比例怎么定？」→ **消融**。
- **易错点与加分表述**：
  **易错**：只说「不会」。**加分**：**双轨评测表**。

### 16. PT 的学习率通常比 SFT 大还是小？

- **30 秒要点**：
  - 一般 **更保守**（数据大、易扰动基座）。
  - 配 **warmup + 余弦**。
- **参考答案（口播）**：
  实践中 CPT 往往用相对保守的学习率，因为更新步数多、数据噪声复杂，过大的学习率容易破坏预训练特征。通常配合线性 warmup 与余弦衰减，并根据验证集与 loss 曲线微调。具体数值依赖模型规模与是否全参，我这边【据实报】，并通过 pilot 对比确认稳定。
- **常见追问与应对**：
  - 「和 SFT 同一数量级可以吗？」→ **看是否全参、是否 LoRA**。
  - 「warmup 比例？」→ **1%～10% step 常见区间**（据实）。
- **易错点与加分表述**：
  **易错**：照搬网上「越大越好」。**加分**：**曲线截图的文字描述**。

### 17. PT 阶段需要 Chat 模板吗？

- **30 秒要点**：
  - **一般不需要**：纯 LM。
  - **注意 EOS**；SFT 再统一模板。
- **参考答案（口播）**：
  CPT 阶段通常是纯文本连续建模，不需要把内容套进对话模板；关键是句子与文档边界、EOS 处理正确，避免拼接错误。进入 SFT 前必须统一 chat_template，否则会出现训练与推理格式不一致。若某些实现把「伪对话」拼进 CPT，那是另一种设定，需要能解释动机，但主流 CPT 不这么做。
- **常见追问与应对**：
  - 「那 tokenizer 特殊 token 呢？」→ **BOS/EOS 策略** 与基座一致。
  - 「多轮知识怎么进 PT？」→ 一般 **不进**，靠 SFT。
- **易错点与加分表述**：
  **易错**：CPT 套模板导致 **分布奇怪**。**加分**：**与官方脚本一致**。

### 18. 如何判断 PT 是否有收益？

- **30 秒要点**：
  - **域内 PPL**、**下游选择题/cloze**、**同量 SFT 消融**。
  - 三者至少两条。
- **参考答案（口播）**：
  我会从三个信号判断：域内验证集 PPL 是否下降；域内选择题或填空类探测任务是否提升；以及固定 SFT 数据与超参，比较「无 CPT vs 有 CPT」谁更快收敛、谁终点更好。若只有 PPL 改善但 SFT 终点没提升，可能语料质量或配比有问题，需要回去清洗或调整混合比例。
- **常见追问与应对**：
  - 「PPL 低但生成差？」→ **语料风格与任务 mismatch**。
  - 「多久评估一次？」→ **按 step 或 epoch 里程碑**。
- **易错点与加分表述**：
  **易错**：只看 loss。**加分**：**探测任务设计**。

## 三、SFT 微调类（10 题）

### 19. SFT 数据格式是什么样的？

- **30 秒要点**：
  - Alpaca：instruction/input/output；ShareGPT：conversations 多轮；与 chat_template 一致；角色清晰。
- **参考答案（口播）**：
  SFT 数据常见两种：一种是 Alpaca 风格，用 instruction、可选 input、output 描述单轮任务；另一种是 ShareGPT 用对话列表存多轮，含 system/user/assistant。无论哪种，训练时都要和 tokenizer 的 chat_template 严格对齐，否则等价于分布错位。实现上通常会把对话渲染成单个 token 序列，并对非 assistant 部分 mask 掉 loss。
- **常见追问与应对**：
  - 模板谁维护？特殊 token？多轮截断？
- **易错点与加分表述**：
  **易错：训练推理模板不一致。加分：能画 **mask 示意图**。**

### 20. SFT 用了多少数据？够吗？

- **30 秒要点**：
  - 报条数、token 估、任务覆盖；质量>数量；LoRA 常见万～十万级（非绝对）。
- **参考答案（口播）**：
  我这边指令数据大概是【N 万条】，覆盖【题型/科室】，并做了去重与分层抽样。是否够主要看验证集与盲评是否还在提升；如果数据少，我会用高质量精选、合成+校验、以及更强正则早停来避免过拟合。对 7B 级模型加 LoRA，社区经验往往在万级到十万级能得到可用效果，但最终以我的评测为准。
- **常见追问与应对**：
  - 数据怎么扩？合成风险？
- **易错点与加分表述**：
  **易错：只报大数不讲覆盖。加分：**难度分层**。**

### 21. Base 模型和 Chat 模型怎么选？

- **30 秒要点**：
  - Chat：格式好、收敛快；Base：可塑性强需更多 SFT。
- **参考答案（口播）**：
  如果目标是快速得到可用对话能力，且 Chat 模型许可证允许微调，通常选 Chat 起点更省事。Base 适合需要强定制话术或担心 Chat 自带对齐叠加的场景，但要准备更多 SFT 数据教会格式与安全。我的选择是【X】，原因是【数据量/风格/许可证】。
- **常见追问与应对**：
  - 继续 DPO 会不会叠加过度对齐？
- **易错点与加分表述**：
  **易错：Base 数据不足硬上。加分：**对比 pilot**。**

### 22. 灾难性遗忘在 SFT 中怎么体现？怎么解决？

- **30 秒要点**：
  - 通用掉点、僵化、过度拒绝；混合 replay、降 LR、控 epoch、LoRA。
- **参考答案（口播）**：
  SFT 过重时可能在通用任务上掉点，或出现只会背训练模板、过度拒答等。缓解包括混入通用指令数据 replay、降低学习率与 epoch、使用 LoRA 限制更新子空间，以及在对齐阶段控制 KL。我们会用 **域外测试集** 监控。
- **常见追问与应对**：
  - LoRA 能完全避免吗？
- **易错点与加分表述**：
  **易错：只说「没观察到」。加分：**前后对比表**。**

### 23. 多轮对话在 SFT 里怎么处理？

- **30 秒要点**：
  - ShareGPT 列表；截断最近 K 轮或 max tokens；loss 常在 assistant。
- **参考答案（口播）**：
  多轮数据用对话列表存储，训练时渲染成序列。截断策略要与推理一致，例如保留最近 K 轮或按 token 预算截断。loss 一般只在 assistant 回合计算，避免学用户说话。长上下文不足时可配合摘要或 RAG。
- **常见追问与应对**：
  - history 太长医学信息丢怎么办？
- **易错点与加分表述**：
  **易错：训练截断与线上一致性没保证。加分：**滑动窗口**策略。**

### 24. SFT 的 loss 是对所有 token 吗？

- **30 秒要点**：
  - 常 mask user/system，只算 assistant CE。
- **参考答案（口播）**：
  标准做法不是对所有 token 都算交叉熵，而是把 prompt 部分 label 设为忽略，只对 assistant 生成部分计算 next-token loss，这样梯度聚焦在「学会如何回答」而不是「预测用户下一句」。若全序列算 loss，任务会偏离。
- **常见追问与应对**：
  - 有哪些实现细节？
- **易错点与加分表述**：
  **易错：与代码不一致。加分：**IGNORE_INDEX** 说明。**

### 25. SFT 需要多少 epoch？

- **30 秒要点**：
  - 常见 1～5；小数据早停；看 val 与生成质量。
- **参考答案（口播）**：
  我通常从【1～3】epoch 起步，结合验证集 loss 与抽样生成质量判断是否继续。数据量小更容易过拟合，需要早停或更强正则。最终以 **盲评** 选 checkpoint，而不是只看训练 loss 最低。
- **常见追问与应对**：
  - 为什么最后几步可能变差？
- **易错点与加分表述**：
  **易错：训满固定 epoch。加分：**checkpoint 扫描**。**

### 26. 学习率、batch、warmup 怎么配？

- **30 秒要点**：
  - LoRA 1e-4～1e-3 扫；累积提有效 batch；warmup+余弦；grad clip；bf16。
- **参考答案（口播）**：
  LoRA 学习率我一般在【范围】网格搜索，配合全局 batch size（用梯度累积凑）。使用 warmup 稳定初期更新，余弦衰减后期精细收敛，并设梯度裁剪防爆。精度用 bf16 或 fp16+loss scale。具体最佳点依赖数据与模型，我以验证集与生成样例为准。
- **常见追问与应对**：
  - 最大 batch 受什么限制？
- **易错点与加分表述**：
  **易错：只抄默认参数。加分：**grad norm 曲线**。**

### 27. Chat 模板写错会怎样？

- **30 秒要点**：
  - 分布偏移、角色乱、推理训练不一致。
- **参考答案（口播）**：
  模板错误会导致模型见到与预训练/指令微调期望不同的 token 分布，表现为角色错乱、空输出、或推理与训练使用不同格式导致效果崩溃。修复方法是训练推理统一模板，并对 encode/decode 写单测样例。
- **常见追问与应对**：
  - 如何排查？
- **易错点与加分表述**：
  **易错：怪数据不查模板。加分：**单测用例**。**

### 28. SFT 如何与后续 DPO 衔接？

- **30 秒要点**：
  - reference 用 SFT；prompt 分布一致；SFT 太差 DPO 难救。
- **参考答案（口播）**：
  DPO 通常以 SFT 模型作为策略起点与 reference，保证能力底座可用；同时 DPO 训练数据的 prompt 分布应尽量被 SFT 覆盖，否则 off-policy 严重。采样温度与格式也要一致。若 SFT 质量差，DPO 会在噪声偏好上学到错误东西，所以我会先提升 SFT 再开 DPO。
- **常见追问与应对**：
  - reference 冻结吗？
- **易错点与加分表述**：
  **易错：prompt 分布错位。加分：**偏好 prompt 与 SFT 重叠率**。**

## 四、LoRA / QLoRA 类（8 题）

### 29. LoRA 的原理是什么？

- **30 秒要点**：
  - 冻结 W；低秩 ΔW=BA；缩放 α/r；有效子空间更新。
- **参考答案（口播）**：
  LoRA 冻结原线性层权重，在旁路加两个低秩矩阵 B∈R^{d×r}、A∈R^{r×k}，用 BA 近似增量更新；前向相当于在原有输出上加一项 scaled BA x。直觉是大幅参数更新往往集中在低维子空间，用很小的 r 就能捕获主要适配方向，从而省显存、加快迭代。
- **常见追问与应对**：
  - 与 Adapter 区别？与 Prefix 区别？
- **易错点与加分表述**：
  **易错：说不清缩放。加分：**参数量估算**。**

### 30. rank（r）怎么选？

- **30 秒要点**：
  - r 大容量大显存大；8/16/32/64 扫；看 val 与下游。
- **参考答案（口播）**：
  r 控制适配器容量。太小可能欠拟合，太大接近全参微调且显存上升。我通常在【8,16,32,64】做网格，结合验证集与生成质量选择；若数据很少，偏小 r 更稳。
- **常见追问与应对**：
  - r 与数据量关系？
- **易错点与加分表述**：
  **易错：固定 64 不解释。加分：**曲线/表格**。**

### 31. alpha（α）一般怎么设？

- **30 秒要点**：
  - 常见 α=2r 或 α=r；与 r 联调。
- **参考答案（口播）**：
  α 与实现里的缩放相关，常见设置是 α=2r 或 α=r，但需要和 r 一起调，因为两者共同影响有效步长。实践以 loss 与生成稳定性为准。
- **常见追问与应对**：
  - α 与学习率耦合？
- **易错点与加分表述**：
  **易错：只调 LR 不调 α。加分：**同 r 不同 α 消融**。**

### 32. LoRA 一般挂在哪些模块？

- **30 秒要点**：
  - 常 q/v；可扩 k/o/MLP；显存紧先 q/v。
- **参考答案（口播）**：
  最常见是先挂 attention 的 q 和 v，性价比好；需要更强容量再扩到 k、o 甚至 MLP。显存紧张时优先 q/v，避免一次全开。
- **常见追问与应对**：
  - 为什么不全挂？
- **易错点与加分表述**：
  **易错：不知道 target_modules。加分：**与论文/库默认对比**。**

### 33. QLoRA 是什么？为什么省显存？

- **30 秒要点**：
  - 4bit 基座 + 高精度 LoRA；基座低位宽。
- **参考答案（口播）**：
  QLoRA 把冻结的基座权重量化到 4bit（如 NF4），在计算时反量化，LoRA 参数仍用 fp16/bf16 训练，从而显著降低显存占用，使得单卡可训更大模型。
- **常见追问与应对**：
  - 速度影响？误差？
- **易错点与加分表述**：
  **易错：以为量化 LoRA。加分：**bitsandbytes 版本**坑。**

### 34. LoRA 和全参数微调对比？

- **30 秒要点**：
  - LoRA：省显存、快迭代、遗忘轻；全参：潜力大、资源高。
- **参考答案（口播）**：
  全参数微调更新所有权重，上限高但对数据与算力要求高，也更容易扰动原能力。LoRA 只训少量参数，迭代快、显存友好，遗忘相对可控，但在极小数据上可能欠拟合。我选 LoRA 是因为【显存/周期】，并用消融对比验证差距可接受。
- **常见追问与应对**：
  - 何时应全参？
- **易错点与加分表述**：
  **易错：贬低 LoRA。加分：**业务指标差距**。**

### 35. LoRA 权重如何合并与部署？

- **30 秒要点**：
  - merge 进基座单文件；或动态 adapter；多租户热切换。
- **参考答案（口播）**：
  推理时可以 merge LoRA 到基座得到单一权重文件，减少加载开销；也可以运行时加载 adapter 实现多版本热切换。合并要注意 dtype 与精度。
- **常见追问与应对**：
  - merge 后能否继续训？
- **易错点与加分表述**：
  **易错：合并版本管理乱。加分：**多 LoRA 路由**。**

### 36. QLoRA 有什么坑？

- **30 秒要点**：
  - 量化误差、驱动兼容、超参需校准。
- **参考答案（口播）**：
  QLoRA 可能遇到量化带来的优化曲面变化，需要重新校准学习率；bitsandbytes 与驱动/CUDA 版本组合也可能踩坑；此外极端 rank 或过大学习率会不稳定。我的做法是先用小数据 sanity check，再扩全量。
- **常见追问与应对**：
  - NF4 vs FP4？
- **易错点与加分表述**：
  **易错：环境报错说不清。加分：**最小复现命令**。**

## 五、RLHF / DPO / 对齐类（12 题）

### 37. RLHF 的完整流程是什么？

- **30 秒要点**：
  - SFT→RM→PPO+KL；RM 学偏好；策略优化防偏移。
- **参考答案（口播）**：
  经典 RLHF 先用 SFT 做行为克隆，再用人类偏好数据训练奖励模型 RM，最后用 PPO 在 RM 打分下优化策略，并用 KL 惩罚贴近 reference 防止分布跑飞与 reward hacking。医疗场景还会在奖励与安全规则上更谨慎。
- **常见追问与应对**：
  - 为何不直接最大化 RM？
- **易错点与加分表述**：
  **易错：步骤顺序说不清。加分：**on-policy 成本**。**

### 38. PPO 在 RLHF 里涉及哪四个模型？

- **30 秒要点**：
  - Policy、Reference、RM、Value(Critic)。
- **参考答案（口播）**：
  四个角色分别是：当前策略模型 policy（actor）生成回答；reference 模型提供 KL 基准；reward model 对回答打分；value/critic 估计状态价值用于优势函数（实现上可能共享部分权重）。训练时四者交互形成 RL 回路。
- **常见追问与应对**：
  - 能否不用 critic？
- **易错点与加分表述**：
  **易错：把 RM 和 policy 混淆。加分：**一次 rollout 描述**。**

### 39. DPO 是什么？相对 RLHF+PPO 的优势？

- **30 秒要点**：
  - 直接偏好优化；隐式奖励；省 RM 在线 RL。
- **参考答案（口播）**：
  DPO 用偏好对直接优化策略相对 reference 的相对似然，不需要在线采样与显式 RM 推理，训练更简洁、调试成本通常低于 PPO 全链路。缺点是偏好数据噪声敏感，且对 reference 与数据覆盖有要求。
- **常见追问与应对**：
  - 劣势？
- **易错点与加分表述**：
  **易错：说 DPO 完全不需要 SFT。加分：**β 作用**。**

### 40. DPO 的损失直觉？

- **30 秒要点**：
  - 拉大 chosen 与 rejected 的 logit 差；β 控距离。
- **参考答案（口播）**：
  直观上 DPO 希望在同 prompt 下提高 chosen 回答的概率、压低 rejected，同时通过 β 限制策略不要离 reference 太远，避免为了赢 rejected 而学坏。
- **常见追问与应对**：
  - 与 Bradley-Terry 关系？
- **易错点与加分表述**：
  **易错：只背公式无直觉。加分：**长度偏见**。**

### 41. ORPO 是什么？

- **30 秒要点**：
  - Odds Ratio Preference Optimization；减少分阶段；以论文实现为准。
- **参考答案（口播）**：
  ORPO 类方法试图把偏好优化更紧密地并入训练目标，减少「先 SFT 再对齐」的分阶段成本，具体目标函数以实现与论文为准。面试中诚实说明你看过的版本即可。
- **常见追问与应对**：
  - 与 DPO 差异？
- **易错点与加分表述**：
  **易错：夸大未读论文。**加分**：结合 **MedicalGPT 是否支持** 据实说。**

### 42. GRPO 的创新点是什么？

- **30 秒要点**：
  - 组内相对优势；弱化 critic；以最新论文为准。
- **参考答案（口播）**：
  GRPO 强调在一组采样回答之间做相对比较来估计优势，减少对单独 critic 的依赖，适合某些可批量采样的场景。医疗上可结合规则检查多样回答。具体细节以论文为准，我避免夸大。
- **常见追问与应对**：
  - 适用场景？
- **易错点与加分表述**：
  **易错：把 GRPO 说成万能。**加分**：**与 PPO 成本对比**。**

### 43. 偏好数据怎么构造？

- **30 秒要点**：

- **参考答案（口播）**：
  常见做法是对同一 prompt 用不同温度或不同模型生成两条回答，再由人工或强规则判定优劣；也可以用现有业务日志里的用户反馈。构造时要去除隐私、去除平局模糊对，并注意长度偏见。
- **常见追问与应对**：
  - 如何保证一致性？
- **易错点与加分表述**：
  **易错：偏好随机。**加分**：**标注指南**。**

### 44. Reward Model 怎么训练？

- **30 秒要点**：
  - Pairwise ranking；chosen 分高于 rejected；防长度偏见。
- **参考答案（口播）**：
  RM 用偏好对训练，让模型对 chosen 打分高于 rejected，损失常用 ranking loss。要防止模型只学「更长更好」，需要归一化、截断或构造长度接近的对。
- **常见追问与应对**：
  - RM 过拟合？
- **易错点与加分表述**：
  **易错：把 RM 当生成模型。**加分**：**校准**。**

### 45. KL 惩罚在 RLHF 里起什么作用？

- **30 秒要点**：
  - 防 hacking；平衡改进与贴近 reference。
- **参考答案（口播）**：
  KL 约束策略不要偏离 reference 太远，减少乱编、谄媚、无意义加长等 reward hacking，同时保留预训练语言能力。系数需要调，过大学不动，过小会跑飞。
- **常见追问与应对**：
  - 与 DPO 中 β 类比？
- **易错点与加分表述**：
  **易错：不知道 KL 针对什么。**加分**：**实际曲线**。**

### 46. PPO 不稳定怎么办？

- **30 秒要点**：
  - 查 LR、clip、KL、reward scale；RM 过拟合；净数据。
- **参考答案（口播）**：
  先检查 PPO 超参与 reward 缩放，再看 KL 是否失控；RM 若过拟合会导致奇怪奖励，需要更多偏好数据或正则；也可加强数据清洗。医疗场景还要排查奖励是否与安全策略冲突。
- **常见追问与应对**：
  - clip 过小？
- **易错点与加分表述**：
  **易错：只调一个旋钮。**加分**：**日志定位**。**

### 47. DPO 需要 SFT 吗？

- **30 秒要点**：
  - 需要；SFT 提供能力与 reference 基座。
- **参考答案（口播）**：
  工程上主流是先 SFT 再 DPO：SFT 让模型具备基本对话与任务能力，DPO 在此基础上做偏好微调；reference 通常取 SFT 模型。跳过 SFT 往往数据要求极高。

  - 能 SFT+DPO 联合吗？
- **易错点与加分表述**：
  **易错：冷启动直接 DPO。**加分**：**reference 选择**。**

### 48. 医疗场景对齐要特别注意什么？

- **30 秒要点**：
  - 安全、合规、标注专业性。
- **参考答案（口播）**：
  医疗对齐要优先处理高风险回答：用药、剂量、诊断结论等要么拒答要么强引用与免责声明；偏好标注尽量由懂医学规范的人或严格指南代理；同时产品层面要有复核机制，而不是只依赖模型。

  - 如何写拒答？
- **易错点与加分表述**：
  **易错：只谈偏好不谈合规。**加分**：**安全评测集**。**

## 八、算法深度追问类（10 题）

### 65. Transformer Self-Attention 的计算复杂度？

- **30 秒要点**：
  - 标准 O(n²d)；长序列瓶颈。
- **参考答案（口播）**：
  自注意力在序列长度 n 上两两交互，复杂度约 O(n²d)，d 为维度；因此长上下文是瓶颈，引出稀疏注意力、线性注意力、FlashAttention 等优化。
- **常见追问与应对**：
  - 线性注意力代价？
- **易错点与加分表述**：
  **易错：只说 O(n²) 无 d。**加分**：**KV cache**。**

### 66. 为什么常用余弦学习率调度？

- **30 秒要点**：
  - 前快后慢、后期精细收敛。
- **参考答案（口播）**：
  余弦衰减让学习率在前期较大快速下降 loss，后期变小精细收敛，曲线平滑；常与 warmup 联用避免一开始破坏预训练特征。
- **常见追问与应对**：
  - 与线性衰减比？
- **易错点与加分表述**：
  **易错：不知道 warmup 配合。**加分**：**曲线草图**。**

### 67. 梯度裁剪（gradient clipping）的作用？

- **30 秒要点**：
  - 限制 grad norm、防爆 NaN。
- **参考答案（口播）**：
  梯度裁剪限制全局梯度范数，防止偶发大梯度导致更新爆炸，在 RNN 时代更重要，但在大模型微调与 RL 里仍然常用作稳定手段。
- **常见追问与应对**：
  - 裁剪阈值怎么定？
- **易错点与加分表述**：
  **易错：与 weight clip 混淆。**加分**：**观察 norm 分布**。**

### 68. FlashAttention 原理直觉？

- **30 秒要点**：
  - 分块 softmax、减 HBM、SRAM 算。
- **参考答案（口播）**：
  FlashAttention 通过分块计算 softmax 与注意力，减少 GPU HBM 读写，把计算留在 SRAM，数值上等价于标准注意力（标准 FA 定义下），从而加速并省显存。
- **常见追问与应对**：
  - FA2？
- **易错点与加分表述**：
  **易错：说改变注意力定义。**加分**：**IO 分析**。**

### 69. LayerNorm 与 RMSNorm 区别？

- **30 秒要点**：
  - LN 减均值除方差；RMSNorm 只用 RMS。
- **参考答案（口播）**：
  LayerNorm 对隐藏向量减均值并除以方差；RMSNorm 省略重中心化，仅按 RMS 缩放，计算更简，常见于部分 LLM。
- **常见追问与应对**：
  - 为何 LLM 爱用 RMSNorm？
- **易错点与加分表述**：
  **易错：混 BN。**加分**：**训练稳定性**。**

### 70. 位置编码 RoPE 的直觉？

- **30 秒要点**：
  - 旋转嵌入相对位置；外推问题。
- **参考答案（口播）**：
  RoPE 通过旋转矩阵把位置信息编码进 QK，强调相对位置关系；长度外推时常讨论缩放与 NTK 等技巧，面试点到即可。
- **常见追问与应对**：
  - ALiBi？
- **易错点与加分表述**：
  **易错：背公式无直觉。**加分**：**长文本实验**。**

### 71. dropout 在微调中还要吗？

- **30 秒要点**：
  - 看验证集；小数据可调低。
- **参考答案（口播）**：
  大模型微调有时仍保留轻微 dropout 或靠 weight decay；若数据很少，过大 dropout 可能伤效果，需要消融。
- **常见追问与应对**：

- **易错点与加分表述**：
  **易错：照搬预训练配置。**加分**：**val 曲线**。**

- **30 秒要点**：

- **参考答案（口播）**：

- **常见追问与应对**：

- **易错点与加分表述**：

### 73. batch norm 为何在 LLM 中不常用？

- **30 秒要点**：
  - 序列与 batch 统计不稳。
- **参考答案（口播）**：
  BN 依赖 batch 统计，序列长度变化大、小 batch 不稳；Transformer 用 LN/RMSNorm 更稳。

### 74. 混合精度训练的数值稳定性？

## 附录 A：口播模板（背诵骨架）

### A1 一分钟项目介绍

- **问题**：【院内导诊 / 医学考试 / 科研辅助】等场景需要更专业、可控的中文医疗对话能力。

- **结果**：在【N 条内部测试】上【指标】提升【x%】，沉淀可复现配置。

### A2 解释 LoRA（30 秒）

- 冻结原权重，用低秩矩阵 BA 近似更新；**r** 控容量，**α** 控步长。
- 显存友好、迭代快，适合领域适配；全参留给大数据与充足算力。

### A3 解释 DPO（45 秒）

- 用偏好对直接优化策略相对 reference 的相对概率；**β** 防止跑飞。
- 相对 PPO+RM：**实现简单、训练稳、资源省**；数据噪声是主要敌人。

## 附录 J：查漏补缺小问（快速过）

### J1 为什么 SFT 常只对 assistant 算 loss？

- 目标是学 **助手回复**；预测用户话会 **偏离任务**。

### J2 为什么 DPO 需要 reference？

- 防止为赢 rejected 而 **极端偏移**；β 调节离 reference 距离。

### J3 ZeRO-3 比 ZeRO-2 多解决什么？

- **参数分片**；显存更低但 **通信更重**。

### J4 FlashAttention 主要优化什么？

- **HBM 访问**；不是改数学定义（标准 FA）。

### J5 医疗场景为何慎用过激 DPO？

- 可能 **过度拒答或风格异常**；需 **安全偏好** 与 **SFT 混合**。

### J6 多轮数据截断策略？

- 最近 K 轮或 max token；与 **推理侧** 一致。

### J7 学习率 warmup 的目的？

- 避免一开始 **过大更新破坏预训练特征**。

### J8 梯度累积影响什么？

- **等效更大 batch**；不改变单步显存（理想情况下）。

### J9 何时考虑 RAG 而非继续 CPT？

- 知识 **更新频繁**、需 **可追溯引用**、CPT 成本过高时。

### J10 RM 的长度偏见怎么缓解？

- 归一化、截断、**成对长度接近**、人工指南。
