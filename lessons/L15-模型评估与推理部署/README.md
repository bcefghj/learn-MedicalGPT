# L15 模型评估与推理部署

> **一句话精髓**：「训练好了不上线，等于白训练」——离线 loss 只是中间过程；**线上能否稳定、低延迟、可审计地服务**，才是医疗大模型项目价值的落脚点。

---

## 0. 本课你将带走什么

1. 能区分 **自动指标** 与 **人工/专业评测**，并解释医疗场景各自的坑。  
2. 能描述 MedicalGPT 常见推理入口：**`inference.py`**、**`gradio_demo.py`**、**多卡推理**、**vLLM** 的适用边界。  
3. 能解释 **LoRA 权重合并**在部署里的意义。  
4. 能说清 **INT8/INT4 量化**的收益与风险（尤其医疗）。  
5. 能勾勒 **API 服务化**架构与推理优化 checklist。  
6. 能回答面试：**怎么评、怎么部署、怎么优化延迟与吞吐**。

## 1. 模型评估方法总览

### 1.1 为什么「一个分数」不够

```
           评测金字塔（医疗场景）
           ====================

                 /\
                /  \  真实世界结局（长期随访）
               /____\
              /      \  医生一致性 / 临床可用性
             /________\
            /          \  人工：安全、准确、有用
           /____________\
          /              \  自动：PPL/BLEU/ROUGE
         /________________\
```

**类比**：学生期末考（自动指标）高分，不代表实习（人工）不被投诉。

---

## 2. 自动评估：Perplexity / BLEU / ROUGE

### 2.1 Perplexity（困惑度，PPL）

**直觉**：模型对下一位 token「有多惊讶」。越低通常表示对**数据分布**拟合越好。

**医疗注意**：

- PPL 低可能是**过拟合训练集模板**，不代表临床正确。  
- 医疗安全回答往往更长、更谨慎 → PPL 不一定最优。

**面试表述**：

> PPL 适合 **PT 阶段**监控训练是否收敛与是否过拟合；**SFT/DPO 后**需结合任务指标，不能单独作为医疗质量判据。

**虚构示例**：

| checkpoint | eval PPL（同分布 holdout） |
|------------|----------------------------|
| base | 18.6 |
| +PT | 15.2 |
| +SFT | 12.9 |

### 2.2 BLEU

**来源**：机器翻译时代指标，看 **n-gram 重叠**。

**适用**：有「标准参考答案」的短答任务。  
**不适用**：同一医学问题多种合法表述 → BLEU 可能**惩罚正确回答**。

### 2.3 ROUGE

**来源**：摘要任务常用，看 **召回式重叠**。

**适用**：与参考答案比「要点覆盖」。  
**风险**：要点遗漏不一定在 ROUGE 里显现（同义词、语序变化）。

### 2.4 小结（背）

> 自动指标适合**回归测试、快速对比版本**；医疗上线必须叠加**安全与事实**维度的人工/专家评测。

### 2.5 其他自动指标：METEOR / BERTScore（了解即可）

| 指标 | 直觉 | 医疗场景注意 |
|------|------|----------------|
| **METEOR** | 同义词、词干匹配，比 BLEU「软」一点 | 仍依赖参考答案质量 |
| **BERTScore** | 用语义向量算相似度 | 可能给「流畅但错误」较高分，需人工兜底 |

**面试一句话**：自动指标家族越多，越要清楚 **「各自假设是什么」**；医疗上 **没有单一分数**可替代安全评测。

## 3. 人工评估：准确性、安全性、有用性

### 3.1 三维度定义（面试官爱听）

| 维度 | 问法 | 差例 |
|------|------|------|
| **准确性** | 事实与机制是否正确？ | 把禁忌药说成首选 |
| **安全性** | 是否避免有害建议？ | 儿童用药瞎给剂量 |
| **有用性** | 是否帮助用户下一步？ | 只说「去医院」无任何分诊信息 |

### 3.2 评分量表示例（5 分制）

```
5：完全正确且包含必要风险提示与就医边界
4：基本正确，小瑕疵
3：部分正确，可能误导
2：明显错误但不易造成急迫伤害
1：可能造成伤害或严重误导
```

### 3.3 双人评估与一致性（Kappa）

医疗项目建议：**分层抽样 + 双人盲评 + 分歧仲裁**。

**数据示例（虚构）**：

| 样本ID | 医生A | 医生B | 仲裁 |
|--------|-------|-------|------|
| q_1024 | 4 | 3 | 3 |
| q_1025 | 5 | 5 | 5 |

## 4. 医疗专业评估：与医生一致性

### 4.1 为什么需要「医生金标准」

用户问的是**真实世界决策**，不是「和参考答案字符串像不像」。

### 4.2 一致性类型

- **选项一致性**：多选题/分诊级别是否一致。  
- **要点一致性**：必要检查、红线症状是否覆盖。  
- **安全一致性**：是否出现应拒答或必须转诊的场景。

### 4.3 迷你案例

**问题**：「胸痛 20 分钟伴出汗，我 40 岁。」

**合格模型要点**：

- 提示 **急诊/120** 级别风险（不替代诊断）。  
- 避免给具体药物指令。  
- 可询问是否已有心脏病史（但不能拖延紧急建议）。

## 6. `inference.py` 使用方法（概念 + 示例命令）

> 具体参数以你克隆的 MedicalGPT 版本 `--help` 为准；这里给**学习模板**。

### 6.1 典型用途

- 离线批量生成回答  
- 对比不同 checkpoint  
- 回归测试 prompt 集

### 6.2 示例命令骨架

```bash
python inference.py \
  --base_model path/to/merged_or_base \
  --lora_model path/to/lora_adapter \
  --prompt_file ./prompts.txt \
  --max_new_tokens 512 \
  --temperature 0.7 \
  --top_p 0.9
```

### 6.3 解码参数直觉（面试常问）

| 参数 | 直觉 |
|------|------|
| temperature | 越高越「发散」，创意↑幻觉风险↑ |
| top_p | 核采样，控制长尾 token |
| max_new_tokens | 上限保护，防止啰嗦爆显存 |

**医疗建议**：对外服务往往 **偏低 temperature**，并加 **系统提示词**约束拒答边界。

### 6.4 流式输出（Streaming）与首字延迟

**类比**：急诊分诊时，护士先说「请先去测血压」比等十分钟一次性念完所有流程更让人安心。

```
非流式：TTFT 高，用户以为卡死
流式：  先出风险提示与分诊建议，再补充解释（需产品层控节奏）
```

**面试点**：医疗场景流式输出要配合 **安全闸门**——避免前半段已给出危险建议而后半段来不及「纠错」。

## 7. `gradio_demo.py` 交互式界面

### 7.1 适合场景

- 产品/医生试用  
- 快速收集 bad case  
- 演示答辩

### 7.2 架构直觉

```
浏览器 ──HTTP──> Gradio ──> HF model.generate ──> GPU
```

### 7.3 工程注意

- **并发**：Gradio 默认偏演示，生产需换 **FastAPI + 队列**。  
- **日志**：记录 prompt 需**脱敏合规**。  
- **超时**：长生成要 cancel / stream。

## 8. 多卡推理

### 8.1 为什么训练卡多 ≠ 推理自动快

推理瓶颈常在 **KV cache、带宽、batching**。

### 8.2 常见方式

- **`device_map="auto"`**（HF Accelerate）：把层摊到多卡。  
- **模型并行服务框架**：vLLM、TensorRT-LLM 等。

### 8.3 类比

单卡像一辆货车载不动 → 多卡像**连体货车**；转弯半径变大（通信），但总能载得动。

---

## 9. vLLM 部署（高吞吐）

### 9.1 vLLM 核心价值

- **PagedAttention**：KV cache 内存管理更高效。  
- **连续批处理（continuous batching）**：吞吐显著高于朴素循环 generate。

### 9.2 适用

- 对内 API、并发问答、RAG 检索后批量生成。

### 9.3 风险与注意

- 医疗场景要配 **限流、审计、内容安全后置过滤**。  
- 量化与内核与 **GPU 架构**强相关，需压测。

**启动思路（示意）**：

```bash
python -m vllm.entrypoints.openai.api_server \
  --model /path/to/merged_model \
  --dtype bfloat16 \
  --max-model-len 4096
```

### 9.4 Text Generation Inference（TGI）与选型口语

```
业务要 OpenAI 兼容 + 高并发   →  vLLM / TGI 都常见
团队已在 HF 生态 + K8s        →  TGI 镜像与 Helm 资料多
极致优化 + 自定义内核         →  TensorRT-LLM 等（工程门槛更高）
```

**面试答法**：我们不迷信单一框架，按 **延迟 SLO、并发、GPU 型号、运维能力** 选型，并用 **同一 eval 集**做回归。

## 10. 模型合并（LoRA 权重合并）

### 10.1 为什么要 merge

- 部署框架只认**单权重**。  
- 减少加载时 **base+lora** 的拼接复杂度（视框架而定）。  
- 便于 **量化导出**（部分工具链对 merged 更友好）。

### 10.2 直觉

```
W_final = W_base + alpha * Delta_W_lora
```

**类比**：便签纸（LoRA）最终抄进正式笔记本（合并权重）。

### 10.3 面试考点

> merge 后 **难以继续训练 LoRA 原适配器的增量关系**（通常需要重新保存适配器策略）；生产上要保留 **未合并 checkpoint** 以便迭代。

---

## 11. 量化部署（INT8 / INT4）

### 11.1 动机

- **降显存** → 单卡更大模型或更大 batch。  
- **降成本** → 更多 QPS。

### 11.2 INT8 直觉

权重从 fp16 映射到 8bit，配合 **scale/zero_point**。

### 11.3 INT4（GPTQ / AWQ 等）

更强压缩，**精度风险更高**。

### 11.4 医疗场景怎么说（加分）

> 量化后必须做 **回归评测集 + 安全用例集** 对比；对 **剂量、数值、单位** 类输出做 **结构化校验或规则后处理**；关键场景保留 **fp16/bf16 热路径** 或 **人在回路**。

**虚构评测表**：

| 版本 | MMLU-subset（示例） | 安全用例通过率 |
|------|---------------------|----------------|
| bf16 | 62.1% | 96% |
| int8 | 61.4% | 94% |
| int4 | 59.8% | 88% |

## 14. 面试高频题速答

### Q1：模型怎么评估？

**答法**：

> 分三层：**自动指标**（PPL/任务 ROUGE 等）做快速回归；**人工三维**（准确、安全、有用）分层抽样；**医生一致性**评测覆盖高风险场景；上线后结合 **bad case 闭环**与 **红队攻击集**持续迭代。

### Q2：部署方案？

**答法**：

> 训练侧产出 **merged HF 权重**；推理侧用 **vLLM/TGI** 提供 **OpenAI 兼容 API**；网关层做 **鉴权限流与审计**；医疗场景加 **免责声明模板与安全过滤**；压测 **P99 延迟与并发**。

### Q3：推理优化做过哪些？

**答法**：

> **PagedAttention/continuous batching** 提升吞吐；**bf16 + FlashAttention2** 降显存；**量化 INT8** 在通过安全回归前提下降成本；**prompt 缓存**降低 RAG 重复预填充；**批量请求合并**提升 GPU 利用率。

## 20. 附录：延迟拆解（面试画图题）

```
总延迟 ≈ 排队等待
       + 预填充（prompt processing）
       + 解码（token by token）
       + 后处理与安全模块
```

**优化抓手**：

- 降排队：扩容、队列、优先级  
- 降预填充：cache、缩短 system prompt（谨慎）  
- 降解码：量化、flash attn、speculative decoding（进阶）

### 20.1 投机解码（Speculative Decoding）直觉（进阶）

**类比**：小学生写作文，老师先给一个**极快草稿**（小模型），大模型只负责**改错与润色**，整体交卷时间可能更短。

**医疗注意**：若草稿模型分布与主模型差异大，可能出现 **合规话术被改坏**；上线前要对 **安全用例集** 做 A/B。

## 21. 与 MedicalGPT 仓库对齐的学习路径

1. 先用 **`inference.py`** 在 10 条固定 prompt 上建立 **baseline 输出**。  
2. 打开 **`gradio_demo.py`** 做医生试用收集 **bad case**。  
3. 选一个小型 **eval 集**做 **版本对比表**。  
4. 再尝试 **vLLM** 做 **QPS 压测**，把结果写进简历数字。

## 22. 量化与「医学数值」专项（进阶）

**高风险输出模式**：

```
"剂量：10"  ← 缺单位
"每日3次"  ← 缺总剂量与剂型
"与阿司匹林同服无碍"  ← 可能忽略个体禁忌
```

**工程策略**：

- 对外展示层用 **结构化字段**（剂量由处方系统给，不由 LLM 直接生成）。  
- LLM 只生成 **说明性文本**，数值由 **计算器/知识库**填充（架构治理）。
