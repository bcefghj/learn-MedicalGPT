# L12 医疗数据集详解

> **一句话精髓**：「数据决定上限，算法决定下限」——再强的架构与优化，也只能在数据所允许的「天花板」之下逼近最优；数据若脏、偏、少，算法再花哨也只是把垃圾包装得更精致。

---

## 0. 本课你将带走什么

1. 能用**类比 + 示意图**向非技术人员解释：为什么医疗大模型项目里「数据工程」往往比「调参」更决定成败。  
2. 能按**训练阶段**（PT / SFT / 偏好对齐）把 MedicalGPT 相关数据集**对号入座**，并说出**量级、格式、典型字段**。  
3. 能复述常见**数据来源构成**（知识图谱、百科、对话等）与**质量评估维度**。  
4. 能设计一版**朴素但合理的数据配比**思路，并应对面试中的「你们用了什么数据」三连问。

## 1. 为什么数据如此重要：从「学霸」到「专科医生」

### 1.1 直觉类比：教材决定考纲

把大模型想象成一个**记忆力极强**的学生：

```
        ┌──────────────────────────────────────────────┐
        │  数据 = 教材 + 习题册 + 考过的真题与评语      │
        └──────────────────────────────────────────────┘
                              │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
         预训练(PT)      指令微调(SFT)    偏好对齐(DPO/RLHF)
         大量阅读        学会按格式作答    学会哪种答法更好
```

- **只有教材（PT）**：他可能「懂很多词」，但考试时不知道要按「问诊—鉴别—建议」的结构写。  
- **加上习题册（SFT）**：他开始会「做题」，但若习题本身有错，他会**错得更自信**。  
- **加上真题评语（偏好数据）**：他学会在多种合法回答里，选更**安全、有用、符合人类预期**的那种。

**结论**：算法与算力决定你能多快、多稳地「逼近」数据所允许的上限；**数据决定这个上限本身有多高**。

### 1.2 医疗场景为什么更「吃数据」

```
   普通开放域                    医疗域
   ---------                    ------
   说错一句 → 可能只是尴尬        说错一句 → 可能误导决策
   用户会自行纠错                 用户往往缺乏医学背景
   「差不多对」常被接受           「差不多对」可能是事故
```

因此医疗项目面试里，面试官往往**先问数据**再问模型：不是刁难，而是行业风险倒逼。

### 1.3 一句工程化表述（可写进简历）

> 我们采用分阶段数据策略：**增量预训练**注入领域知识与术语分布，**SFT**固化对话与指令遵循能力，**DPO/偏好数据**降低有害与幻觉回答的概率；全链路配套**去重、质检、配比与敏感信息治理**。

### 2.1 240 万条中文医疗数据集（`shibing624/medical`）

**定位**：偏「领域语料池」，常用于**增量预训练**或作为**知识型监督样本**的来源之一（具体用法看你选择的训练脚本与数据管线）。

**为什么要单独提它**：它把模型的「词分布」往**医学中文**拉近——类似让学霸突然开始只读医学系教材，说话会少很多网络梗，多很多解剖、药理、诊疗路径用语。

**数据量级（记忆口径）**：约 **240 万条**（面试时说「两百万级中文医疗文本样本」即可；若你实际子集不同，务必以你训练日志与数据统计脚本为准）。

**可能的记录形态（示例字段，教学演示）**：

```json
{
  "text": "慢性胃炎患者常表现为上腹隐痛、餐后饱胀。幽门螺杆菌感染是重要病因之一。",
  "source": "medical_corpus_subset",
  "lang": "zh"
}
```

> 面试提示：不要背「字段名」，要背**用途**——「提升医学中文语言模型能力、术语与共现统计」。

### 2.2 22 万条中文医疗对话数据集

**定位**：更贴近「问答/问诊」交互，适合作为 **SFT** 的重要组成，让模型学会**轮次、礼貌、追问、解释**等对话行为。

**示例（高度简化，演示多轮结构）**：

```json
{
  "conversations": [
    {"role": "user", "content": "我这两天头疼，还有点恶心，需要去医院吗？"},
    {"role": "assistant", "content": "头痛伴恶心原因很多。若出现剧烈头痛、视物模糊、肢体无力、高热或颈部僵硬，应尽快急诊。否则可先观察并记录诱因与伴随症状，必要时门诊神经科或内科就诊。本回答仅供参考，不能替代医生面诊。"}
  ]
}
```

**实际数据示例（单轮 instruction 风格，也常出现）**：

```json
{
  "instruction": "患者出现持续性胸痛，伴出汗，可能提示什么？",
  "input": "男，55岁，高血压病史。",
  "output": "需要高度警惕急性冠脉综合征等危急情况，建议尽快急诊评估（心电图、心肌酶等）。以下为科普信息，不能替代急诊处理。"
}
```

### 2.3 预训练数据：16GB 中英文语料、中文维基百科

**类比**：这是「通识教育」。没有它，模型可能医学术语很溜，但**逻辑、写作、常识推理**会偏科。

**为什么要中英文一起**（常见工程理由，面试可答）：

1. **论文与指南**：大量医学证据与综述以英文发表，双语语料有助于术语对齐与跨语言概念迁移（视模型与训练目标而定）。  
2. **推理与代码/结构化表达**：英文语料往往带来更稳定的「结构化说明」风格（非绝对，但是常见经验）。  
3. **避免过窄域导致的能力坍塌**：纯医疗语料训练容易让模型在域外任务上退化（具体程度与配比相关）。

**中文维基**的价值：相对干净的**百科叙述**，对「定义—机制—分类」这种写作方式很友好。

**虚构但合理的规模示例（帮助你建立数量级直觉）**：

| 子集 | 规模（示例） | 作用 |
|------|--------------|------|
| 中文通用新闻/网页清洗语料 | ~10GB（示例） | 语言能力底座 |
| 英文通用语料 | ~6GB（示例） | 双语与证据类文本补充 |
| 中文维基（清洗后） | ~数百万到千万级 tokens（视清洗而定） | 定义型知识叙述 |

> 重要：面试时说「大约十几 GB 级预训练语料 + 医疗增量」比报一个精确到小数点后两位的数字更可信，除非你手上有**真实统计**。

### 2.4 SFT 数据：ShareGPT、Belle、Alpaca 等

**共同点**：把模型从「会续写」变成「会按要求生成」——**指令遵循 + 对话格式**。

**ShareGPT 风格（多轮）**核心结构：

```json
{
  "conversations": [
    {"from": "human", "value": "帮我解释一下 II 型糖尿病的发病机制。"},
    {"from": "gpt", "value": "II 型糖尿病主要与胰岛素抵抗和相对胰岛素分泌不足有关……"}
  ]
}
```

**Alpaca/Belle 风格（单轮为主）**核心结构：

```json
{
  "instruction": "用三句话解释疫苗如何训练免疫系统。",
  "input": "",
  "output": "疫苗通常含有灭活或减毒病原体或其片段……"
}
```

**医疗项目里的关键**：SFT 样本里最好系统性加入**免责声明**、**就医建议**、**信息不足时追问**，否则模型学会「像搜索引擎一样确诊」，风险极高。

### 2.5 偏好数据：DPO-En-Zh-20k-Preference、`oasst1` 等

**DPO 数据在学什么**：不是「正确答案唯一」，而是**在两种回答之间学会偏好**——更像老师批改作文时说「这篇更好，因为更安全/更清晰/更完整」。

**极简偏好对示例（字段名因数据集而异，理解语义即可）**：

```json
{
  "prompt": "我发烧 38.5℃，需要吃抗生素吗？",
  "chosen": "发热不等于细菌感染，抗生素需医生评估后使用。可先对症处理并观察，若持续高热或出现气促、皮疹等应尽快就医。",
  "rejected": "吃阿莫西林吧，一天三次，连吃一周。"
}
```

**`oasst1` 一类开放助理偏好/对话数据**：常用于提升** helpful、honest、harmless** 三角里的 helpful 与表达多样性；医疗项目通常会**二次清洗**并**降低非医疗胡编样本权重**。

## 3. 数据来源拆解：79万 / 36万 / 79万 的「拼图感」

很多医疗语料库会宣称由多条渠道融合而来。下面用**结构化拼图**帮助记忆（数字为你题目给定口径，实际项目请用你自己数据血缘统计验证）：

```
                 医疗数据「三大拼图」（示例口径）
                 ================================

              ┌─────────────────┐
              │ 知识图谱 79万条  │  关系三元组/结构化问答转文本
              └────────┬────────┘
                       │
   ┌───────────────────┼───────────────────┐
   │                   │                   │
   ▼                   ▼                   ▼
实体-关系-属性      症状-疾病链路        药品-禁忌-相互作用
（适合 PT/SFT）    （适合问诊助理）      （适合用药安全提示）
```

| 来源类型 | 量级（本课给定口径） | 典型内容形态 | 训练阶段常见用途 |
|----------|----------------------|--------------|------------------|
| 知识图谱衍生 | 约 **79 万条** | 「疾病A 的常见症状包括 …」「药物B 禁忌 …」 | PT / SFT（结构化转自然语言） |
| 在线医疗百科 | 约 **36 万条** | 百科条目、疾病介绍、检查说明 | PT / SFT（定义型叙述） |
| 医疗对话 | 约 **79 万条** | 问诊记录、医患 QA、健康咨询 | **SFT 主力**（对话行为） |

**类比**：  
- 知识图谱像「医学知识卡片」——准但可能干巴巴。  
- 百科像「教科书段落」——解释性强。  
- 对话像「门诊录音文字稿」——真实但噪声大。  

**工程结论**：三类数据要**互相补齐**：对话教「怎么说」，图谱/百科教「说什么更靠谱（但仍需质检）」。

## 4. 数据质量评估维度：别只盯着「条数」

### 4.1 维度一：准确性（Accuracy / Correctness）

- **看什么**：事实是否与权威指南/教材一致，是否存在过时建议。  
- **医疗特有风险**：把「可能性」说成「确定性」。

**迷你案例（错误类型示例）**：

```
用户：脚扭了肿了怎么办？
错误回答：立刻热敷促进血液循环。（急性期通常更谨慎的是按 RICE 原则，是否热敷需分阶段）
```

### 4.2 维度二：完备性（Completeness）

- **看什么**：关键追问是否缺失（年龄、孕哺、过敏史、时长、诱因）。  
- **好的助理风格**：信息不足 → **先问关键问题** → 再给分层建议。

### 4.3 维度三：多样性（Diversity）

- **看什么**：科室覆盖、人群覆盖、罕见病占比、问题表述口语化程度。  
- **过低的后果**：模型过拟合模板回答，泛化差。

### 4.4 维度四：一致性（Consistency）

- **看什么**：同一事实在不同样本中是否打架；中英文术语是否对齐。  
- **常见来源**：不同爬虫源、不同编辑规范拼接导致。

### 4.5 维度五：安全性（Safety）与合规（Compliance）

- **看什么**：隐私信息、广告引流、非法诊疗承诺、恐吓营销。  
- **策略**：脱敏、过滤、红队样本对抗、拒答模板。

### 4.6 可量化指标（面试可以提「我们有表」）

| 指标 | 含义（口语） | 例子 |
|------|--------------|------|
| 去重率 | 重复越多，浪费算力且过拟合 | SimHash 近似重复占比 |
| 毒性/辱骂率 | 对话数据常见 | 分类器抽检 |
| PII 命中率 | 隐私风险 | 正则 + NER |
| 平均长度 / token 分布 | 是否极端偏长偏短 | 直方图 |
| 语言混杂率 | 中英夹杂是否异常 | 语言识别 |

## 5. 数据配比策略：不是「一锅炖」，而是「调鸡尾酒」

### 5.1 为什么要谈配比

```
  数据配比失衡的典型后果
  ======================

  医疗对话太多、百科太少  →  会「很会聊」，但定义题胡编概率上升
  百科太多、对话太少      →  会「像维基」，但不会追问与共情表达
  偏好数据过脏            →  DPO 学坏偏好，越对齐越危险
```

### 5.2 朴素但好用的起点（教学用「示例配方」）

> 下列比例**不是标准答案**，只给你面试时展示「我有方法论」。真实比例要靠验证集与人工评测迭代。

| 阶段 | 建议思路（示例） |
|------|------------------|
| PT | 通用语料为主体（例如 70%–90%），医疗语料逐步加量（10%–30%），避免能力坍塌 |
| SFT | 医疗指令/对话为主体；混入少量通用指令防遗忘（常见经验：5%–20% 量级视遗忘情况） |
| DPO | 数据量通常远小于 SFT，但要求更高：干净、对比鲜明、覆盖高风险场景 |

**渐进式训练日程类比**：

```
周一-周三：健身练下肢（PT 打底座）
周四-周五：练动作技术（SFT 学格式）
周六：教练纠正发力点（DPO 修偏好）
```

### 5.3 「课程学习」式配比（Curriculum）

1. 先短文本、强模板、低噪声 → 学会遵循格式。  
2. 再加入长文本、复杂多轮 → 提升上下文能力。  
3. 最后加入高风险场景（用药、儿童、孕哺）→ 强化拒答与转诊提示。

## 6. 面试高频题：标准答法（建议背框架，不背空话）

### Q1：你们项目用了什么数据？

**答法模板（STAR 的 S/T）**：

> 我们数据按三阶段组织：**增量预训练**使用通用中英文语料与中文维基，叠加 **shibing624/medical** 两百万级中文医疗文本；**SFT**使用约二十万级中文医疗对话，并融合 ShareGPT/Belle/Alpaca 风格的指令数据以强化指令遵循；**对齐阶段**引入 DPO-En-Zh-20k-Preference 与 oasst1 等偏好数据，重点优化安全性与回答结构。数据来源上融合知识图谱、医疗百科与对话三类渠道，并配套去重与质检。

### Q2：数据量多少？

**要点**：

- **分阶段报**：PT 用 GB/token、SFT 用条数、DPO 用「偏好对数量」。  
- **说明是否去重后统计**：面试官更信「去重后」。  
- **说明子集**：全量下载 ≠ 全量训练。

### Q3：怎么保证数据质量？

**要点清单（背 7 条就够）**：

1. **清洗**：HTML 标签、乱码、广告、重复。  
2. **去重**：精确 + 近似（MinHash/SimHash）。  
3. **质检规则**：医学禁用表述、绝对化承诺、隐私模式。  
4. **抽样人工审核**：按科室/场景分层抽样。  
5. **模型辅助过滤**：用小模型做预标注与风险分类。  
6. **版本管理**：数据快照与可复现训练配置绑定。  
7. **红队与对抗样本**：专门攻击「瞎开药、瞎诊断」。

## 10. 实战附录：如何用「小脚本思维」统计你自己的数据（面试加分）

面试官喜欢听的不是「我下载了数据集」，而是「我**量过**数据集」。下面给出一个**教学用**统计清单，你可用 pandas / jq / 纯 Python 实现：

```
┌─────────────────────────────────────────────────────────────┐
│  数据集体检表（建议每次训练前导出成 CSV / Markdown 存档）      │
├──────────────────┬──────────────────────────────────────────┤
│ 文件数 / shard 数 │ 避免遗漏子目录                            │
│ 总条数            │ raw vs dedup 两套数字都要                  │
│ 字段缺失率        │ instruction/input/output 谁经常为空        │
│ 平均字符数        │ 过短可能是垃圾，过长可能是爬取噪声            │
│ 语言比例          │ zh/en/mixed                               │
│ 重复率 Top        │ 最常见模板是否刷榜                          │
│ 高风险关键词命中   │ 「包治」「根治」「100%」等营销话术          │
└──────────────────┴──────────────────────────────────────────┘
```

**类比**：这就像体检报告里的「血压、血脂、血糖」——单项异常不一定致命，但**组合异常**就要警惕上训练。

**示例结论写法（可贴在实验记录里）**：

> 去重后 SFT 样本 **182,430** 条；其中 `input` 为空的占比 **61%**（符合 Alpaca 风格预期）；近似重复簇 **3.2%** 已剔除；高风险营销词命中 **0.07%** 已人工复核。

## 11. 数据合规与授权：比「能下载」更重要

### 11.1 类比：食材进货台账

```
        餐厅后厨                         训练数据
        --------                         --------
   供应商许可证                    数据集 License / 合同条款
   冷链追溯码                      数据血缘（从哪爬、谁标注）
   过期食材处理                    过期指南/错误医学内容的下线流程
```

**面试加分句**：

> 我们区分 **「可研究」与「可商用」**；对外服务使用的数据链路必须具备 **授权证明 + 留存策略 + 二次分发边界** 的书面记录。

### 11.2 常见坑（背 4 条）

1. **爬虫协议**：`robots.txt` 与网站 ToS 是否允许机器学习用途。  
2. **匿名化再识别风险**：罕见病 + 地理位置 + 时间线可能重新指向个人。  
3. **跨域混用**：对话数据来自 A 平台，百科来自 B 源，**免责声明与责任边界**要分开写清。  
4. **开源协议传染性**：某些「开源语料」对衍生模型权重是否有限制（视具体 License 而定，需法务核对）。

### 11.3 工程落地最小集合

```
[ ] 数据清单 CSV：名称、版本、条数、License、负责人
[ ] 训练配置绑定：data_manifest_sha256
[ ] 定期复核：指南更新 → 语料下线或重标注
```

## 12. 数据划分与泄漏：别让评测「作弊」

### 12.1 什么是泄漏（Leakage）

**类比**：考试前把真题原封不动背进考场——分数虚高，真上临床就露馅。

常见泄漏源：

- **同一条问答**经去重不彻底，同时出现在 train 与 eval。  
- **同一患者轨迹**被切成多条样本，跨集合重复。  
- **模板化爬取**导致「前半句在 train、后半句在 eval」的软泄漏。

### 12.2 划分策略 ASCII

```
           全量原始池
                │
        ┌───────┴───────┐
        v               v
   按「文档/会话 ID」划分      绝不允许随机行级打散后同会话跨集合
        │
        v
   train / val / test（比例按资源，医疗建议保留足够 test）
```

### 12.3 面试标准答法

> 我们按 **会话级或文档级** 划分；eval 集 **冻结**且 **版本化**；上线前用 **时间切分** 或 **新收集 holdout** 模拟真实漂移。

## 14. 偏好数据「坏例」与修复：DPO 也会学坏

### 14.1 坏偏好对示例（虚构）

```json
{
  "prompt": "胃溃疡能喝浓茶吗？",
  "chosen": "一般不建议空腹大量饮用浓茶；具体结合症状与用药请咨询医生。",
  "rejected": "绝对不能喝，喝了会胃癌。"
}
```

**问题**：`rejected` 虽然更差，但若数据集中大量存在 **恐吓式、绝对化** 的 `chosen`，模型会学会「吓人式合规」——用户体验差，也不符合医学沟通规范。

### 14.2 修复思路

- **偏好维度拆解**：安全 / 清晰 / 共情 / 结构 —— 不要只标「谁更长」。  
- **tie-break**：两人标注不一致 → 进入仲裁桶或丢弃。  
- **医疗专桶**：用药、儿童、孕哺 **单独增标**。

## 15. 数据与目标的「对齐矩阵」（渐进式自检）

用一张表把「我有什么数据」映射到「我想优化什么行为」：

```
                    想优化的行为
                 准确事实  对话礼貌  拒答边界  多轮追问
  PT 百科/图谱      ★★★      ★        ★        ☆
  SFT 指令          ★★      ★★★      ★★       ★★
  DPO 偏好          ★       ★★      ★★★      ★★
```

**读法**：★ 多表示「通常更直接相关」；空白表示「别指望单靠这一类数据解决」。

## 16. 面试官反追问：你怎么接招

| 面试官追问 | 推荐接法 |
|------------|----------|
| 「你们数据有没有标注成本？」 | 分层：规则自动标 + 医生抽检 + 高风险全人工。 |
