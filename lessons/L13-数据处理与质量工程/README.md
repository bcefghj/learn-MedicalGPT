# L13 数据处理与质量工程

> **一句话精髓**：「Garbage in, garbage out」——脏数据进，幻觉与偏见出；医疗场景里，这不仅是模型效果问题，更是**安全与合规**问题。

## 1. 为什么数据处理是「隐形的主角」

### 1.1 类比：餐厅后厨

```
前厅（推理服务）   : 用户看到端上来的菜
后厨（数据处理）   : 洗菜、切配、冷链、留样、废油处理

用户只会骂「菜难吃」
但真正出事常在「没洗干净」
```

大模型训练也一样：Loss 降不下去、对齐越训越怪、评测忽高忽低，**回头一查**经常是数据管线有洞。

### 1.2 医疗域的「放大效应」

| 普通文本噪声 | 医疗文本噪声的后果 |
|--------------|-------------------|
| HTML 残留 | 模型学会输出 `<div>` 式回答 |
| 乱码 | 术语断裂，诱导胡编 |
| 旧指南未更新 | 「推荐疗法」过时，误导风险 |
| 广告软文 | 「包治」话术被学会 |

## 2. 数据清洗流程：推荐的标准流水线

### 2.1 总览 ASCII 流程图

```
     原始数据（多源异构）
              │
              v
      ┌───────────────┐
      │ 1) 合规与授权  │  合同/许可证/爬虫协议
      └───────┬───────┘
              v
      ┌───────────────┐
      │ 2) 敏感信息治理 │  脱敏、删除、哈希化
      └───────┬───────┘
              v
      ┌───────────────┐
      │ 3) 格式解析    │  HTML/PDF/JSON/CSV
      └───────┬───────┘
              v
      ┌───────────────┐
      │ 4) 规范化      │  全角半角、空白、标点
      └───────┬───────┘
              v
      ┌───────────────┐
      │ 5) 语言检测    │  过滤非目标语言大块噪声
      └───────┬───────┘
              v
      ┌───────────────┐
      │ 6) 去重        │  精确 + 近似
      └───────┬───────┘
              v
      ┌───────────────┐
      │ 7) 质量规则    │  长度、脏词、医学禁用表述
      └───────┬───────┘
              v
      ┌───────────────┐
      │ 8) 分层抽样质检│  人工 + 模型辅助
      └───────┬───────┘
              v
      ┌───────────────┐
      │ 9) 分桶与导出  │  PT / SFT / DPO JSONL
      └───────────────┘
```

### 2.2 每一步「面试怎么说」

- **合规与授权**：我们有权训练吗？能否二次分发？（企业内部数据尤其关键）  
- **敏感信息治理**：电话、身份证、病历号、精确地址。  
- **格式解析**：PDF 表格转文本容易错位，要专门校验。  
- **规范化**：避免同一药品名十种写法导致学习困难。  
- **语言检测**：医疗对话里夹杂大量英文缩写是正常的，阈值要调。  
- **去重**：重复样本会放大错误答案的梯度信号。  
- **质量规则**：把「确定式诊断」改成「需面诊评估」类模板（属于标签策略+清洗策略结合）。  
- **分层抽样**：儿科/精神科/用药问答分别抽，避免只抽到内科。  
- **分桶导出**：同一字段 schema 固定，训练脚本才稳定。

## 3. 文本预处理：去重、去噪、格式标准化

### 3.1 去重（Deduplication）

**精确去重**：对整段文本做 hash（MD5/SHA256）。适合抓「完全复制粘贴」。

**近似去重（Near-dup）**：抓「改了两个标点」的复制。

```
  文本 A: "阿司匹林可抑制血小板聚集，用于心血管事件预防。"
  文本 B: "阿司匹林可以抑制血小板聚集，用于预防心血管事件。"

  精确去重：不认为重复
  近似去重：很可能认为重复（视阈值）
```

**常见方法（面试提名词即可）**：

- **SimHash**：海明距离判相似。  
- **MinHash + LSH**：大规模语料常用。  
- **嵌入聚类**：成本高，适合精品集。

**实际数据示例（去重前后）**：

| 阶段 | 条数（示例） |
|------|--------------|
| raw | 1,000,000 |
| 精确去重后 | 930,000 |
| 近似去重后 | 885,000 |

### 3.2 去噪（Denoising）

**典型噪声类型**：

1. **HTML/XML 标签**：`<p>`、`&nbsp;`。  
2. **导航栏/页脚**：「首页｜登录｜注册」。  
3. **OCR 错误**：「氨氯地平」→「氨氯地平」（形近字）。  
4. **乱码与替换符**：``。  
5. **异常长度**：单条 1 个字符或 50KB 纯重复。

**规则示例（伪代码思想）**：

```
IF 可打印字符比例 < 0.85 → drop
IF 连续重复字符长度 > 30 → drop
IF URL 占比 > 0.3 → review 或 drop
```

### 3.3 格式标准化（Normalization）

**中文场景常做**：

- 全角半角统一（括号、逗号、百分号）。  
- 空白折叠：多空格 → 单空格。  
- 统一医学单位：`mg` 与 `毫克` 是否映射（要谨慎，先统计再决策）。  
- 数字标准化：`３８．５℃` → `38.5℃`（示例）。

**类比**：像把不同医院的化验单统一成同一套表头，后面统计才靠谱。

## 4. Tokenization：什么是分词，BPE 是什么

### 4.1 分词在训练里到底服务谁

模型看到的是 **token id 序列**，不是人类眼中的「词」。

```
人类:  "患者出现高血压"
模型:  [101, 8721, 3301, 8892, 102]   ← token id（示例）
```

**Tokenizer**负责：`字符串 → token 序列 → 再可逆还原（近似）`。

### 4.2 BPE（Byte Pair Encoding）直觉

**类比**：英文像「字母乐高」，中文像「部首乐高」。BPE 会从高频子串开始，把常见片段合并成一个新的 token。

```
起始:  低 / 血 / 糖 / 危 / 象
合并: 「低血」常一起出现 → 新 token
再合并: 「血糖」→ 新 token
...
```

**好处**：

- 词表有限，**未登录词（OOV）**减少。  
- 能同时处理英文子词与中文多字词。

**面试一句话**：BPE 是**数据驱动的子词切分**，在压缩序列长度与词表大小之间折中。

### 4.3 医疗文本对 Tokenizer 的「刁难」

```
长化学名、基因名、药品商品名：
"苯磺酸氨氯地平片" 可能被切成很多子词 → 训练效率下降、语义碎片化
```

## 5. 词表扩充：为什么中文医疗常要扩词表

### 5.1 不扩词表会怎样

- 长术语被切碎 → **同样概念占用更多 token** → 同样上下文窗口能装的医学信息变少。  
- 推理时更容易出现「半截词」相关的表面连贯但事实错误（非唯一原因，但是常见现象之一）。

### 5.2 类比：字典页码

**小通用字典**：每个医学长词要拆成很多常见字解释，读起来啰嗦。  
**医学附录**：给长词单独词条，阅读更省页数。

### 5.3 MedicalGPT 的 `extend_vocab` 思路（概念层）

> 具体函数名与脚本以你克隆的 MedicalGPT 版本为准；这里讲**工程语义**。

典型步骤：

1. **统计**医疗语料中的高频 n-gram / 词组。  
2. **过滤**：过短、过泛、脏词剔除。  
3. **加入 tokenizer 词表**：扩展 `vocab.txt` 或等价结构。  
4. **调整模型嵌入层**：`resize_token_embeddings(new_size)`。  
5. **（可选）**对新嵌入做初始化：均值初始化或从子词嵌入组合初始化。  
6. **继续训练**：PT 或 SFT 让新 token 获得语义。

**风险提醒（面试加分）**：

- 词表盲目变大 → **嵌入矩阵变大** → 显存与训练成本上升。  
- 扩词表后若不做充分训练，新 token 可能是**随机坑位**。

## 6. 数据格式转换：instruction 与 ShareGPT

### 6.1 instruction 格式（Alpaca 家族）

```json
{
  "instruction": "解释 CT 和 MRI 的适用场景差异。",
  "input": "患者怀疑颅内病变。",
  "output": "CT 对急性出血、钙化敏感，速度快；MRI 对软组织对比更好……以下信息仅供参考。"
}
```

**特点**：  
- `input` 可为空。  
- 适合单轮任务、易于批处理。

### 6.2 ShareGPT 格式（多轮）

```json
{
  "conversations": [
    {"from": "human", "value": "我体检尿酸高，需要吃药吗？"},
    {"from": "gpt", "value": "是否用药取决于数值、合并症、痛风发作史等，需要医生评估……"},
    {"from": "human", "value": "数值是 480，没有痛风。"},
    {"from": "gpt", "value": "仍建议由医生结合肾功能、血压、用药史等综合判断……"}
  ]
}
```

### 6.3 转换策略（工程上常写个小脚本）

**ShareGPT → Alpaca（有损）**：只取最后一轮问答 → **丢失上下文**，谨慎使用。  
**Alpaca → ShareGPT（无损结构）**：单轮变一对 `human/gpt`。

**面试考点**：多轮数据flatten成单轮时，模型可能学不到「追问与承接」。

## 7. 数据增强技术：医疗场景的「红线与绿线」

### 7.1 绿线（相对安全、常用）

- **同义改写**：把「头疼」与「头痛」在低风险句子里替换（需规则约束）。  
- **语序微调**（谨慎）：仅对说明性文本，不对用药指令。  
- **模板化重述**：同一事实，多种组织方式（仍需人工抽检）。  
- **回译（back-translation）**：中→英→中，提升表述多样性（成本高，医疗要审）。

### 7.2 红线（高风险，默认不做或强人工审核）

- **随意改剂量/频次**：`每日一次` ↔ `每日三次` 是事故级增强。  
- **把「可能」改「一定」**：会放大模型武断。  
- **伪造患者信息**：合规雷区。

### 7.3 增强的「量」要服从不确定性原则

```
        数据增强强度
            │
            ├─ 科普解释类：可增强空间较大
            │
            └─ 诊疗决策类：增强≈制造新标签，必须医生审核
```

## 8. 数据质量评估：四维 + 可落地指标

### 8.1 准确性（Accuracy）

- **抽检**：与指南核对。  
- **交叉验证**：同一问题多数据源是否一致。  
- **指标示例**：人工标注「事实错误率」= 错误条 / 抽检条。

### 8.2 完备性（Completeness）

- **字段完整率**：`output` 是否空、`conversations` 是否缺 `gpt`。  
- **信息是否够答**：用户只给「头疼」，标准答案是否先追问。

**示例**：

```
不完备回答：你这就是偏头痛。（武断）
完备回答：需要了解持续时间、部位、伴随症状……（追问）
```

### 8.3 多样性（Diversity）

- **n-gram 分布**：是否只剩 20 个模板。  
- **科室覆盖表**：内科占比是否 90%。

### 8.4 一致性（Consistency）

- **术语表**：「心梗」与「心肌梗死」是否映射一致。  
- **冲突检测**：样本 A 说禁忌 X，样本 B 说可用 X。

## 9. 敏感信息处理（PII / PHI）

### 9.1 常见敏感项

- 姓名、手机号、身份证号、住址、病历号、住院号。  
- 精确时间线 + 罕见病 → 有时也能**间接识别**个人。

### 9.2 处理策略

```
┌────────────┬──────────────────────────────┐
│ 删除       │ 最强，信息损失最大            │
│ 掩码       │ 138****1234                   │
│ 合成替换   │ 用虚构但一致的身份（研究常用） │
│ 哈希化     │ 可追踪不可读（企业内部）      │
└────────────┴──────────────────────────────┘
```

### 9.3 面试标准表述

> 我们在数据入湖阶段做 **PII 检测**，对命中字段执行 **掩码或删除**；训练集导出使用 **不可逆脱敏**；训练环境与对外演示数据 **隔离**；保留 **数据版本号**以便审计。

## 10. 面试高频题：怎么答才像「干过」

### Q1：你们数据处理流程是什么？

**推荐结构**：合规 → 脱敏 → 解析 → 规范化 → 语言检测 → 去重 → 规则过滤 → 分层质检 → 分桶导出 →（可选）增强 → 版本冻结。

### Q2：如何保证数据质量？

**答 8 件事**（与 L12 呼应但更工程）：

1. **Schema 校验**：JSON 字段必须符合约定。  
2. **静态规则**：长度、脏词、禁用承诺。  
3. **近似去重**：降低重复背书错误。  
4. **分层抽检**：按科室/任务类型。  
5. **双人复核**：高风险子集。  
6. **模型预标注**：只做筛子，不做最终裁判。  
7. **红队样本库**：专门收集「诱导瞎开药」的 prompt。  
8. **数据血缘**：每个 shard 可追溯到源。

## 11. 迷你案例：一条脏数据如何「毒害」SFT

**原始样本（虚构）**：

```json
{
  "instruction": "小孩发烧怎么办？",
  "input": "",
  "output": "吃阿司匹林退烧，剂量按成人减半。"
}
```

**问题点**：

- 儿童发热处理 **不应默认推荐阿司匹林**（瑞氏综合征风险等，需专业判断）。  
- 「成人减半」是危险简化。

**清洗后（示例方向）**：

```json
{
  "instruction": "小孩发烧怎么办？",
  "input": "年龄 3 岁，体温 38.6℃，精神尚可。",
  "output": "儿童发热原因很多，是否需要退热药、用哪种药、剂量多少，应由医生根据年龄体重与病因评估。家长可观察精神状态与脱水表现，若嗜睡、呼吸急促、持续高热或家长担忧，应及时就医。以下为科普信息。"
}
```

**启示**：医疗数据清洗不是「去 HTML」这么简单，更是**临床安全红线治理**。

## 16. BPE 训练过程：从「字符」到「合并表」（渐进理解）

### 16.1 类比：社团合并

想象学校里有上千个**单字社团**，每天统计「哪两个字最常手拉手出现」，就把它们**合并成一个新社团名**。反复合并，直到社团数量达到目标词表大小。

### 16.2 极简算法骨架（面试讲步骤即可）

```
词表初始化：每个字符是一个 token（或 byte 级）
repeat:
  1) 在全体语料上统计相邻 token 对频率
  2) 取频率最高的 pair (A, B)
  3) 新建 merge：AB → 新符号
  4) 更新语料中的切分结果
until 达到预设词表大小或增益饱和
```

### 16.3 与 WordPiece 的对比（选择题/面试常混）

| 维度 | BPE（GPT/LLaMA 系常见） | WordPiece（BERT 系常见） |
|------|-------------------------|---------------------------|
| 合并依据 | 高频 **字节对/子词对** | 似然提升（最大化语料概率） |
| 输出 | merge 规则表 | ## 子词标记等 |
| 直觉 | 「谁常一起出现就合并」 | 「怎样合并让整体概率更高」 |

**医疗提示**：同一套 **预训练 tokenizer** 与 **增量语料** 若不匹配，会出现**碎片化术语**——这正是词表扩充动机之一。

## 17. `extend_vocab` 落地检查清单（与 Hugging Face 生态对齐）

下面用**伪代码心智**描述常见流程（函数名以你项目为准）：

```text
1) tokenizer = AutoTokenizer.from_pretrained(base)
2) new_tokens = load_medical_phrases_filtered(...)   # 去重、长度、脏词过滤
3) num_added = tokenizer.add_tokens(new_tokens)
4) model.resize_token_embeddings(len(tokenizer))
5) 可选：对新增 embedding 行做均值初始化
6) 继续 PT 或 SFT，让新 token 获得上下文语义
```

**面试高频追问**：`resize_token_embeddings` 之后要不要调学习率？

> 常见做法：**整体略降 lr 或对新增 embedding 行稍大 lr（需谨慎）**；更稳的是先 **小规模 warmup** 观察 loss 与生成质量。

## 19. 清洗规则引擎：从「写 if」到「可配置」

### 19.1 为什么需要规则引擎

纯人工盯百万条不现实；纯模型过滤又可能**误杀**医学缩写。工程上常采用 **规则优先 + 模型复核**。

### 19.2 规则分层 ASCII

```
  Layer1  硬规则（长度、编码、非法字符）
      │
      v
  Layer2  词典规则（广告词、违禁承诺、非法引流）
      │
      v
  Layer3  近似重复（SimHash/MinHash）
      │
      v
  Layer4  小模型分类器（辱骂、色情、非医疗灌水）
      │
      v
  Layer5  人工抽检队列（只看不规则命中样本）
```

### 19.3 实际样例：一条规则 YAML（教学虚构）

```yaml
rules:
  - name: no_absolute_cure
    pattern: "(包治|根治|百分百治愈|无效退款)"
    action: quarantine
  - name: pediatrics_high_risk
    if_contains: ["小孩", "儿童", "婴儿"]
    extra_check: must_have_disclaimer_or_triage
```

## 20. 数据增强「对照实验」思维（写进实验笔记）

**虚构 ablation（只教你怎么记表）**：

| 设置 | SFT 验证集（人工分） | 备注 |
|------|----------------------|------|
| 无增强 | 3.85 | baseline |
| 同义改写（低风险域） | 3.91 | 多样性↑ |
| 回译（中→英→中） | 3.88 | 成本高 |
| 乱改剂量（错误增强） | 3.12 | **负例：千万别上线** |

**结论写法**：医疗增强必须 **分层评估**；任何改变**数值/剂量/频次**的增强默认 **禁止**。
